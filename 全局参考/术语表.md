## A. K8s 核心对象（Kubernetes 基本积木）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **Kubernetes（K8s）** | 容器编排系统（希腊语"舵手"，K 与 s 之间省去 8 字母） | Google 开源的容器集群管理系统，负责调度、扩缩、自愈。本项目目标实操环境是基于 kubeadm 自建的 5 节点离线集群（v1.29.15） |
| **kubeadm** | K8s 官方集群部署工具 | 用一条 `kubeadm init` + `kubeadm join` 拉起生产级集群。本项目集群就是 kubeadm 部署（非 K3s/kind），升级见 `版本升级/` |
| **Pod（容器组）** | K8s 最小调度单元 | 一个或多个共享网络/存储的容器组成的运行实例；真正跑业务的是 Pod，不是容器 |
| **ReplicaSet（副本集）** | 副本控制器 | 保证指定数量的 Pod 副本始终运行，副本数不够就新建、多了就删 |
| **Deployment（部署）** | 无状态工作负载控制器 | 管理 ReplicaSet 的版本，支持滚动更新与回滚；本项目 90% 业务用 Deployment 发布 |
| **StatefulSet（有状态集）** | 有状态工作负载控制器 | 每个 Pod 有固定名字+独立 PVC（如 `db-0`/`db-1`），适合数据库等需稳定身份的服务；`数据库部署/` 用得到 |
| **DaemonSet（守护进程集）** | 节点级常驻控制器 | 每个（或指定）节点上跑且只跑一个副本，常用于日志/监控 agent；Falco、node-exporter、Cilium agent 都是 DaemonSet |
| **Namespace（命名空间）** | 资源隔离分区 | 把集群逻辑切成多块（如 `monitoring`/`argocd`/`tomcat-prod`），用于权限与资源配额隔离 |
| **ConfigMap（配置映射）** | 配置字典 | 把非机密的配置（如环境变量、配置文件）与镜像解耦，Pod 以文件或变量挂载 |
| **Secret（秘密）** | 敏感配置 | 存密码/Token/证书等机密，默认 base64 编码（非加密）。明文存 Git 不安全 → 引出 Sealed Secrets / Vault |
| **Job / CronJob** | 任务 / 定时任务 | Job 跑一次完即止；CronJob 按 crontab 周期跑（如 KEDA 之外的定时批处理） |
| **CRD** | Custom Resource Definition（自定义资源定义） | 让 K8s 认识新资源类型，是"扩展 K8s"的机制；ArgoCD 的 `Application`、Argo Rollouts 的 `Rollout` 都是 CRD |
| **Operator** | 运维自动化控制器 | 把人肉运维逻辑写成代码（自定义控制器），Prometheus Operator、cert-manager、Velero 都靠它 |

## B. 工作负载与调度（怎么跑、跑多少、跑在哪）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **HPA** | Horizontal Pod Autoscaler（水平 Pod 伸缩） | 按 CPU/内存/自定义指标自动增减 Pod 副本数；`K8s基础/三基石` 与 `弹性伸缩/` 都涉及 |
| **VPA** | Vertical Pod Autoscaler（垂直 Pod 伸缩） | 改单个 Pod 的 CPU/内存配额（本项目未主用，KEDA 补位） |
| **KEDA** | Kubernetes Event-Driven Autoscaling（事件驱动伸缩） | 基于外部事件（消息队列长度、Cron 时间表）伸缩，支持到 0；`弹性伸缩/` 用 Cron ScaledObject 实现工作日 9–18 扩缩 |
| **ResourceQuota（资源配额）** | 命名空间资源上限 | 限制某 Namespace 能用多少 CPU/内存/PVC，防单业务吃满集群；`可靠性保障/` |
| **LimitRange（限制范围）** | Pod/容器默认资源约束 | 给未声明资源的容器设默认 request/limit，防"裸奔"吃资源；`可靠性保障/` |
| **PDB** | Pod Disruption Budget（ Pod 中断预算） | 规定"同时能挂几个 Pod"，节点维护/驱逐时保证可用性；`可靠性保障/` 设 `minAvailable=2` |
| **PriorityClass（优先级类）** | 调度优先级 | 资源紧张时高优先级 Pod 先调度、低优先级被挤占；`可靠性保障/` 设 2 个类 |
| **Taint / Toleration（污点/容忍）** | 节点排斥机制 | 给节点打"污点"排斥 Pod，Pod 声明"容忍"才能上去；master 用 `NoSchedule` 挡业务 Pod |
| **Affinity（亲和性）** | 调度偏好 | 控制 Pod 倾向/排斥落在哪些节点或和哪些 Pod 同框；Chaos Mesh、Falco 用 `nodeSelector`/`NoSchedule` 限定只在 worker |

## C. 网络与流量（请求怎么进来、Pod 怎么互通）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **CNI** | Container Network Interface（容器网络接口） | K8s 网络插件标准；本项目数据面由 **Cilium（eBPF）** 接管，Calico 作回退备援 |
| **Cilium** | 基于 eBPF 的 CNI（专有名词，无缩写全称） | 用 eBPF 实现网络+安全+可观测，替代 iptables；本项目生产 CNI，模式 VXLAN |
| **Calico** | 另一主流 CNI（专有名词） | 用 BGP/iptables 做网络策略；本项目作 Cilium 回退资料（`Calico-配置指南/`） |
| **eBPF** | extended Berkeley Packet Filter（Linux 内核可编程框架） | 在内核安全挂载小程序，实现高性能网络/观测/安全，无需改内核源码；Cilium/Hubble/Falco 的底座 |
| **Hubble** | Cilium 的可观测组件 | 基于 eBPF 展示服务间流量、DNS、网络策略命中，数据喂给 Grafana；`eBPF-可观测性/` |
| **Service（服务）** | 稳定的访问入口 | 给一组 Pod 一个固定虚拟 IP+DNS，屏蔽后端 Pod 漂移；三类：ClusterIP / NodePort / LoadBalancer |
| **ClusterIP** | 集群内部服务 IP | 仅在集群内可达，默认类型 |
| **NodePort** | 节点端口暴露 | 在每个节点开放 30000–32767 端口，外部经 `<节点IP>:<端口>` 访问；`K8s基础/三基石` |
| **LoadBalancer** | 负载均衡型服务 | 调云厂商 API 建外部 LB，每 Service 独占公网 IP；本项目离线环境少用 |
| **Ingress（入口）** | 七层流量路由规则 | 定义"域名→Service→Pod"的 HTTP/HTTPS 路由；`K8s基础/三基石` + `流量入口/` |
| **Ingress Controller（入口控制器）** | Ingress 的执行者 | 监听 Ingress 变更并渲染反向代理配置，如 nginx-ingress；`流量入口/` |
| **Gateway API** | K8s 新一代流量治理 API | 比 Ingress 更灵活（按角色拆分 Gateway/HTTPRoute、支持权重金丝雀）；`流量入口/` 用 Envoy Gateway 实现 80/20 分流 |
| **Envoy Gateway** | 基于 Envoy 的 Gateway API 实现 | 本项目 `流量入口/` 采用的 Gateway API 控制器，版本 v1.2.0 |
| **HTTPRoute** | Gateway API 的路由规则 | 定义流量如何从 Gateway 路由到后端 Service，支持 header/权重匹配 |
| **NetworkPolicy（网络策略）** | 微隔离规则 | 类似"Pod 防火墙"，控制 Pod 间谁能访问谁；Cilium/Calico 均支持 |
| **VXLAN** | 虚拟扩展局域网（Overlay 隧道） | Cilium 在本项目用的封装模式，跨节点 Pod 通过隧道互通 |

## D. 存储（数据怎么持久化）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **PV** | PersistentVolume（持久卷） | 集群中的一块存储（NFS/SSD/local），由管理员建或 StorageClass 动态建 |
| **PVC** | PersistentVolumeClaim（持久卷声明） | Pod 申请存储的"订单"，如"我要 5Gi 的盘"；绑定到 PV 后挂载使用 |
| **StorageClass（存储类）** | 存储类别 | 定义存储"型号"（NFS/SSD/local）+ Provisioner，实现动态供给；`env.sh` 里 `STORAGE_CLASS` 变量控制 |
| **Provisioner（供给器）** | 动态建盘程序 | 按 StorageClass 自动创建 PV；本项目用 NFS Provisioner 实现 RWX 动态供给（`存储管理/`） |
| **RWX / RWO** | ReadWriteMany / ReadWriteOnce | 存储访问模式：多节点同时读写 / 单节点读写；Harbor、MinIO 等需 RWX |
| **MinIO** | 兼容 S3 的对象存储 | 本项目用它当 S3 后端，承接 Loki、Tempo 的日志/链路存储，及 Velero 备份；部署在 `monitoring` 命名空间 |
| **S3** | 简单存储服务（AWS 对象存储协议） | 业界对象存储标准接口；Loki/Tempo/Velero 都认 S3 协议，MinIO 即私有 S3 实现 |

## E. 弹性与安全治理（准入与防护）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **RBAC** | Role-Based Access Control（基于角色的访问控制） | 控制"谁(Subject) 能对 什么资源(Resource) 做什么动作(Verb)"；`SA` 是 Pod 的身份 |
| **SA** | ServiceAccount（服务账户） | Pod 访问 K8s API 时的身份；ArgoCD/Falco 等都要配 SA + RBAC |
| **Admission Webhook（准入_webhook）** | 准入拦截器 | Pod 创建前经"准入控制"拦截校验/改写；Validating（校验，如 Kyverno 拒违规）、Mutating（改写，如 Istio/Linkerd 注入 sidecar） |
| **Kyverno** | K8s 策略即代码引擎（专有名词） | 用 YAML 写策略做准入控制：禁 latest 标签 / 限可信仓库 / 要求探针 / 禁特权；`策略即代码-Kyverno/` |
| **Policy（策略）** | Kyverno 规则集 | 一条策略=一组规则（match→validate/mutate/generate）；本项目 4 条已实测拦截 |
| **OPA / Gatekeeper** | 另一策略引擎（Open Policy Agent） | Kyverno 的同类竞品，用 Rego 语言写策略；本项目选 Kyverno（YAML 更友好） |
| **Sidecar（边车）** | 辅助容器 | 注入到业务 Pod 旁做代理/采集（Istio/Linkerd 的 proxy、OTel 采集器）；占用额外资源 |
| **mTLS** | mutual TLS（双向传输层安全） | 客户端和服务器端互相验证证书，服务网格默认开启；Linkerd/Istio 的 STRICT mTLS 已验证 |
| **Probe（探针）** | 健康检查机制 | Liveness（活否，挂了重启）/ Readiness（就绪否，挂了摘流量）/ Startup（启动慢保护）；Kyverno 强制要求 |

## F. 持续交付 CI/CD & GitOps（代码怎么变上线）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **CI** | Continuous Integration（持续集成） | 代码提交后自动构建/测试/打镜像，保证每次提交可部署；本项目用 Jenkins / GitLab CI |
| **CD** | Continuous Delivery/Deployment（持续交付/部署） | 把通过 CI 的产物自动部署到目标环境；本项目用 ArgoCD（GitOps）做 CD |
| **GitOps** | Git 即真相源的交付范式 | 变更只改 Git 里的 YAML，工具自动把集群同步成 Git 声明状态；本项目核心范式 |
| **Jenkins** | 老牌 CI 工具（专有名词） | 方案1 的 CI 引擎，靠 Pipeline 脚本构建镜像并触发 ArgoCD |
| **GitLab CI** | GitLab 内置 CI | 方案2 的 CI 引擎，靠 `.gitlab-ci.yml` + GitLab Runner 构建；Demo 为 Java/Tomcat |
| **GitLab Runner** | GitLab CI 的执行器 | 真正跑 CI 任务的 agent；注册到 GitLab 后认 `.gitlab-ci.yml` |
| **ArgoCD** | GitOps 持续部署工具（Argo 家族） | 监听 Git 变更→自动 sync 到 K8s；核心是 CRD `Application`；`方案1/2` 共用 |
| **Application（应用）** | ArgoCD 的核心 CRD | 定义"哪个 Git 仓库的哪个路径 → 部署到哪个集群的哪个命名空间" |
| **sync（同步）** | ArgoCD 的状态对齐动作 | 把 Git 期望状态 apply 到集群；`auto-sync` 自动、`manual` 手动 |
| **auto-sync / selfHeal / prune** | 自动同步 / 自愈 / 修剪 | 三个关键模式：变更自动同步 / 集群被手改后自动修复 / Git 删了集群也删 |
| **outOfSync** | 状态不一致标识 | ArgoCD 标集群实际与 Git 声明不符的状态，需 sync 拉齐 |
| **Argo Rollouts** | 渐进式发布工具（Argo 家族） | 支持金丝雀(Canary)与蓝绿(BlueGreen)发布，带 Analysis 自动判稳；方案3 |
| **Canary（金丝雀）** | 渐进放量 | 逐步把流量从旧版切到新版（如 10%→30%→100%），异常即停；源自"矿工伤前先放金丝雀测毒气" |
| **BlueGreen（蓝绿）** | 双版本切换 | 新版(绿)全部署好再一次性切流量，旧版(蓝)留作秒级回滚；`方案3` |
| **Analysis / Experiment** | 发布判稳任务 | Rollouts 在放量途中跑的健康检查（查错误率/延迟），不过就继续、不过就回滚 |
| **Kustomize** | K8s 原生配置管理 | base + overlays 管理多环境差异，无需模板语言；`方案3` 用 overlays 分 3 环境 |
| **Helm** | K8s 包管理器 | 把一组 YAML 打成 Chart 支持变量/依赖；中间件多用 Helm 装 |
| **Pipeline（流水线）** | CI 多阶段任务 | 把构建/测试/推送拆成 Stage 顺序或并行执行 |
| **DIND** | Docker-in-Docker | 在容器内跑 Docker 守护进程，CI 里用它 `docker build` 打镜像 |
| **webhook（通用）** | 回调钩子 | 广义指事件触发 HTTP 回调；CI 发版成功回调、告警发钉钉都算 |

## G. 可观测性三支柱（系统健康怎么看）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **三支柱** | Metrics + Logs + Traces | 可观测性三大来源：指标(有什么问题)/日志(为什么)/链路(哪里)；本项目监控三支柱即此 |
| **Metrics（指标）** | 时序数值 | CPU/内存/QPS 等随时间变化的数字；Prometheus 采集 |
| **Logs（日志）** | 文本记录 | 应用/容器输出文本，出问题查"为什么"；Loki 聚合 |
| **Traces（链路）** | 分布式追踪 | 一次请求跨多个服务的完整路径，定位"哪里慢"；Tempo 承接 |
| **Prometheus** | 指标采集与告警（专有名词） | 拉(Pull)模式按时抓 /metrics 端点；本项目 28 targets ALL UP |
| **Prometheus Operator** | 用 CRD 管 Prometheus | 用 `ServiceMonitor` 等 CRD 声明采集目标，免手写 scrape 配置 |
| **ServiceMonitor** | Prometheus 采集目标声明 | 声明"哪个 Service 的 /metrics 要被抓"；本项目改造后 28 targets 全 UP |
| **NodeExporter** | 节点指标导出器 | 暴露机器级指标（CPU/磁盘/网络）给 Prometheus |
| **AlertManager** | 告警管理器 | 对 Prometheus 告警分组/抑制/静默，经邮件/Webhook/钉钉发出 |
| **Grafana** | 可视化面板（专有名词） | 把 Prometheus/Loki/Tempo 数据画成 Dashboard，也是统一查询入口 |
| **Loki** | 轻量日志聚合（Grafana 出品） | 类 PromQL 语法查日志，存储后端已切 MinIO(S3)；`可观测性/02-logs` |
| **Promtail** | Loki 的日志采集端 | 类似 node-exporter 但采日志，把容器日志发给 Loki |
| **Tempo** | 分布式追踪后端（Grafana 出品） | 存 trace，支持 OTLP；本项目单体 3.0 + MinIO(S3)；`可观测性/05-otel` |
| **OpenTelemetry（OTel）** | 可观测性数据采集标准 | 厂商中立的采集/导出框架；Collector 收口 :4317/4318 → Tempo |
| **OTLP** | OpenTelemetry Protocol（遥测协议） | OTel 的传输协议，gRPC(4317)/HTTP(4318) 两种端口 |
| **SkyWalking / OAP** | 国产 APM（应用性能监控） | 早期链路方案，后端 OAP 做聚合拓扑；本项目已卸载，改 Tempo |
| **Elasticsearch（ES）** | 搜索/日志存储引擎 | 曾作 SkyWalking 存储，3 节点；2026-07-09 已卸载释放资源 |
| **LGTM** | Loki+Grafana+Tempo+MinIO 全栈 | 可观测性"全家桶"代称：日志(Loki)+面板(Grafana)+链路(Tempo)+对象存储(MinIO 作 S3)全 S3 化 |
| **Dashboard（仪表盘）** | 可视化面板 | Grafana 里一组图表；本项目有一套 Cluster Overview + 业务面板 JSON |

## H. 安全与 DevSecOps（四层闭环：扫描→准入→运行时→供应链）

| 术语 | 中文全称 | 一句话解释与本项目角色 |
|------|----------|------------------------|
| **TLS** | Transport Layer Security（传输层安全） | HTTPS 的加密基础，保护客户端-服务端通信 |
| **cert-manager** | 证书自动化工具（Jetstack） | 自签 CA→ClusterIssuer→自动签发/续期 TLS；本项目 4 SAN 域名 |
| **ClusterIssuer** | 集群级证书签发者 | cert-manager 的"CA 机构"抽象，Ingress 引用它拿证书 |
| **Sealed Secrets** | Git 安全存 Secret 方案（Bitnami） | `kubeseal` 加密 Secret 后可安全进 Git，集群内自动解密；`密钥管理/` |
| **Vault** | 集中密钥库（HashiCorp） | 统一存动态密钥（KV v2），本项目 dev 模式根 token=root；`密钥进阶/` |
| **ESO** | External Secrets Operator（外部密钥算子） | 把 Vault 等外部密钥自动同步成 K8s Secret，应用无感知；`密钥进阶/` |
| **SecretStore / ExternalSecret** | ESO 的两个 CRD | SecretStore 声明"从哪取"（连 Vault），ExternalSecret 声明"取哪个、同步成啥 Secret" |
| **Trivy** | 漏洞扫描器（Aqua） | 扫镜像/Repo/文件系统漏洞；Harbor 集成它做镜像扫描 |
| **cosign** | 镜像签名工具（Sigstore） | 用本地私钥给镜像签名，证明"出处可信"；`供应链安全/` |
| **SBOM** | Software Bill of Materials（软件物料清单） | 列出镜像里所有组件/依赖，出事能快速定位受影响版本；用 `syft` 生成 |
| **syft** | SBOM 生成工具（Anchore） | 扫描镜像产出 SBOM（SPDX/ CycloneDX 格式） |
| **verifyImages** | Kyverno 验签策略 | 镜像进群前验签名，未签名一律拒（本项目 `imageReferences=192.168.1.61:5000/*`）；`供应链安全/` |
| **Falco** | 运行时威胁检测（Sysdig） | 基于 eBPF 监控异常进程/提权/读敏感文件；`运行时安全/Falco/`，modern_ebpf |
| **ChaosMesh** | 混沌工程平台（专有名词） | 主动注入 Pod/网络/压力故障验证系统韧性；`混沌工程-ChaosMesh/`，限 worker |
| **Velero** | 集群备份恢复（专有名词） | 把 K8s 资源+PV 备份到 S3( MinIO)，可整集群恢复；`备份容灾/` |
| **DevSecOps** | 安全左移的研发运维范式 | 把安全嵌进 DevOps 全流程；本项目四层闭环：Trivy 扫描 + Kyverno 准入 + Falco 运行时 + cosign 供应链验签 |

## J. 本项目习惯用语（只在本项目语境成立）

| 术语 | 本项目含义 |
|------|-----------|
| **env.sh** | 统一环境配置（all-in-one），所有部署脚本 `source` 它拿到 Harbor IP/域名/StorageClass 等变量 |
| **deploy-helper.sh** | envsubst 模板渲染函数库，提供 `apply_template()` 等通用部署函数，把 `${VAR}` 渲染成实际值 |
| **envsubst** | 变量替换工具，把 YAML 里的 `${VAR}` 换成 env.sh 的实际值，实现"换集群只改一个文件" |
| **方案1 / P1** | Jenkins + ArgoCD 方案，Demo 为 Python SnowNLP 情感分析 |
| **方案2 / P2** | GitLab CI + ArgoCD 方案，Demo 为 Java Tomcat 应用 |
| **方案3 / P3** | Argo Rollouts 灰度方案（金丝雀+蓝绿），复用方案2 的 Git 仓库 |
| **K8s 三基石** | HPA（弹性）+ PV/PVC（存储）+ Ingress（入口），入门三条核心能力 |
| **监控三支柱** | Metrics(Prometheus) + Logs(Loki) + Traces(Tempo)，可观测性三大支柱（原 SkyWalking 已换 Tempo） |
| **聚焦模式** | 全家桶缩容到 0、只留控制面+核心组件的省资源运行态，用于专攻 23~31 增强项 |
| **target / instance** | Prometheus 采集目标；一个 target 通常对应一个 Pod/Service 的 /metrics |
| **job（Prometheus）** | 一组同类型 scrape targets 的采集任务 |
