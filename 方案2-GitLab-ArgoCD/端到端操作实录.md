> 本文档记录了从已有 K8s 集群 + Harbor（私有镜像仓库） + ArgoCD 基础上，新增 GitLab + GitLab CI Runner，实现 Tomcat Java Web 应用从代码 Push → GitLab CI 构建 → Harbor → ArgoCD 同步的全过程。
>
> **零占位符，零省略**：每个命令都可以直接复制执行（替换环境变量后即可）。
>
> 📁 **独立配置文件目录**: [配置文件/](配置文件/) — 所有 YAML/Java/Shell 配置已提取为独立文件，带详尽中文注释。

## 一、基础设施部署

### 1.1 部署 GitLab CE

```bash
# 将所有配置复制到操作目录
cd F:\项目管理2026\项目实战\方案2-GitLab-ArgoCD

# 如果 StorageClass 不是 local-path，先替换:
# sed -i 's|# storageClassName: local-path|storageClassName: 你的SC|g' 配置文件/gitlab/gitlab-deploy.yaml

# 部署 GitLab
kubectl apply -f 配置文件/gitlab/gitlab-deploy.yaml

# 确认资源创建
kubectl get pvc,pod,svc -n gitlab
# 期望: 3 个 PVC, 1 个 Pod (Pending → Running), 1 个 Service
```

### 1.2 等待 GitLab 就绪

```bash
# ★ 首次启动约需 3-5 分钟，请耐心等待
kubectl wait --for=condition=ready pod -l app=gitlab -n gitlab --timeout=600s

# 实时查看日志（确认没有 Error）
kubectl logs -n gitlab deployment/gitlab -f
# 看到 "GitLab is ready!" 表示启动完成
```

### 1.3 获取 GitLab Root 密码

```bash
# 密码存储在 initial_root_password 文件中（24小时内有效）
kubectl exec -n gitlab deployment/gitlab -- cat /etc/gitlab/initial_root_password

# 输出示例:
# Password: aB3xK9mP2qR7vW...
#
# ★ 请记录下来！这是唯一获取密码的机会！
```


```bash

# Admin Area → CI/CD → Runners → New instance runner
# 复制 Registration token
```


## 八、CI Variables 配置（详细说明）

### 8.1 必需的 Variables

在 GitLab 项目 → Settings → CI/CD → Variables：

```
HARBOR_USER = admin
  Type: Variable
  Protected: No
  Masked: No

HARBOR_PASS = Harbor12345
  Type: Variable
  Protected: No
  Masked: Yes

GITLAB_TOKEN = glpat-xxxxxxxxxxxx
  Type: Variable
  Protected: No
  Masked: Yes
```

### 8.2 GITLAB_TOKEN 的权限要求

创建 Token 时至少需要勾选：
- `api`
- `read_repository`
- `write_repository`

否则 Stage 5 (gitops-push) 无法 push 回 GitLab。

### 8.3 测试 Variables 是否正确

在 GitLab CI/CD → Pipelines → Run pipeline 手动触发一次，观察 Stage 3 (docker-push) 是否能成功登录 Harbor。

## 九、端到端验证

### 9.1 完整"改代码 → 部署"流程

```bash
cd /tmp/tomcat-repo

# 1. 修改代码（例如修改 index.jsp 中的版本号显示）
sed -i 's|1.0.0|2.0.0|g' src/main/webapp/index.jsp

# 2. 提交并推送
git add src/main/webapp/index.jsp
git commit -m "feat: bump version to 2.0.0"
git push origin main

# 3. GitLab CI 自动触发 Pipeline

# 4. 等待 Pipeline 完成（约 3-5 分钟）
# 在 GitLab Web UI 观察

# 5. Pipeline 完成后，等待 ArgoCD 自动同步（约 1-3 分钟）

# 6. 验证新版本已上线
kubectl port-forward -n tomcat-demo svc/tomcat-app 8080:8080 &
curl -s http://localhost:8080/ | grep "2.0.0"
# 应该看到页面版本已更新为 2.0.0
```

### 9.2 验证滚动更新零宕机

```bash
# 在一个终端持续请求
while true; do
  curl -s http://localhost:8080/health | grep version
  sleep 1
done

# 在另一个终端触发新构建
# 观察请求返回的版本号变化，应该平滑过渡无中断
```

### 9.3 验证 ArgoCD Self-Heal

```bash
# 手动修改 K8S 资源（模拟配置漂移）
kubectl scale deployment tomcat-app -n tomcat-demo --replicas=1

# 等待 30 秒观察
kubectl get pods -n tomcat-demo
# ArgoCD 应该自动将 replicas 恢复为 2 (selfHeal=true)

# 查看 ArgoCD 日志
kubectl logs -n argocd deployment/argocd-application-controller | grep tomcat-app
```
