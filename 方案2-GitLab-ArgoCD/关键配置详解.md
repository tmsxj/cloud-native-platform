## 一、.gitlab-ci.yml（CI 流水线）

**文件位置**：`配置文件/tomcat-app/.gitlab-ci.yml`

> 📄 **独立配置**: [配置文件/tomcat-app/.gitlab-ci.yml](配置文件/tomcat-app/.gitlab-ci.yml)

### 为什么用 9 个 Stage？

```
3 Stage 方案:   Build → Docker → Deploy   (太粗，无质量门禁)
5 Stage 方案:   Build → Docker Build → Push → Update GitOps → Git Push (缺测试和多环境)
9 Stage 方案:   unit-test → checkstyle+spotbugs → build → docker-build
                → docker-push → deploy-dev → deploy-staging → deploy-prod
                ✅ 包含完整质量门禁 + 多环境渐进式部署
```

### 9 个 Stage 全景

| Stage | Job | 用途 | 产出 |
|-------|-----|------|------|
| `test` | `unit-test` | Maven 单元测试 | JUnit 报告 + 覆盖率 |
| `static-scan` | `checkstyle` | 代码风格检查 | checkstyle-result.xml |
| `static-scan` | `spotbugs` | 静态缺陷扫描(可失败) | spotbugsXml.xml |
| `build` | `build` | Maven 编译 WAR | target/*.war artifact |
| `security-scan` | `docker-build` | Docker 镜像构建 | /tmp/image.tar artifact |
| `docker-push` | `docker-push` | 推送到 Harbor | 版本标签 + latest |
| `deploy-dev` | `deploy-dev` | DEV 部署清单更新 | k8s/dev/deployment.yaml |
| `deploy-staging` | `deploy-staging` | STAGING 部署清单更新 | k8s/staging/deployment.yaml |
| `deploy-prod` | `deploy-prod` | PROD 部署(手动触发) | k8s/prod/deployment.yaml |

### Stage 1: unit-test

```yaml
unit-test:
  stage: test
  image: ${REGISTRY}/library/maven:3.9-eclipse-temurin-17-alpine
  script:
    - mvn test -B
  artifacts:
    when: always
    reports:
      junit: target/surefire-reports/TEST-*.xml
```

**设计意图**：
- 在所有构建之前先跑单元测试，**快速失败**原则
- JUnit 报告集成到 GitLab CI UI，可直接在 Merge Request 中查看
- `when: always` 确保即使测试失败也产出报告

### Stage 2: static-scan（并行质量门禁）

```yaml
checkstyle:
  stage: static-scan
  script:
    - mvn checkstyle:check -B

spotbugs:
  stage: static-scan
  allow_failure: true   # SpotBugs 可失败，不阻塞流水线
  script:
    - mvn spotbugs:check -B
```

**设计意图**：
- checkstyle + spotbugs **同一 stage 并行执行**，节省时间
- SpotBugs `allow_failure: true`：告警不阻塞，适合渐进式引入
- 质量门禁先于编译，**越早发现问题成本越低**

### Stage 3-5: Build → Docker Build → Push

**为什么分离这三个 Stage？**
- Build 失败 → Maven 编译问题
- Docker Build 失败 → Dockerfile 问题
- Push 失败 → Harbor 认证/网络问题
- 分离后故障定位精确到具体环节

### YAML Anchor 模板（deploy 复用）【核心】

```yaml
.deploy_template: &deploy_template
  image: ${REGISTRY}/library/alpine:latest
  before_script:
    - apk add --no-cache bash git openssh-client
  script:
    - git config user.email gitlab-ci@cicd.local
    - git config user.name GitLab-CI
    - git remote set-url origin http://oauth2:${GITLAB_TOKEN}@gitlab.gitlab.svc.cluster.local/root/tomcat-app.git
    - git checkout -B main                    # ★ 解决 detached HEAD
    - git fetch origin main && git rebase origin/main  # ★ 先同步再修改
    - sed -i "s@image: .*@image: ${IMG}:${TAG}@" ${DEPLOY_FILE}
    - grep image: ${DEPLOY_FILE}
    - git add ${DEPLOY_FILE}
    - git commit -m "[skip ci] deploy ${ENV} ${TAG}" || echo no-changes
    - git push origin main
```

**设计意图**：
- YAML Anchor (`<<: *deploy_template`) 让 3 个 deploy job 共用同一套逻辑
- `DEPLOY_FILE` 和 `ENV` 通过 `variables` 差异化配置
- `[skip ci]` 前缀防止部署提交触发新一轮 Pipeline（无限循环问题）
- `git checkout -B main`：解决 GitLab CI 默认的 detached HEAD 状态
- `git fetch + rebase`：**先同步远程变更再修改文件**，避免推送冲突

### 三个 Deploy Job 的差异

```yaml
deploy-dev:
  <<: *deploy_template
  needs: [docker-push]              # 依赖 docker-push 完成
  variables:
    DEPLOY_FILE: k8s/dev/deployment.yaml
    ENV: DEV

deploy-staging:
  <<: *deploy_template
  needs: [deploy-dev]               # 依赖 DEV 部署完成（串行）
  variables:
    DEPLOY_FILE: k8s/staging/deployment.yaml
    ENV: STAGING

deploy-prod:
  <<: *deploy_template
  needs: [deploy-staging]           # 依赖 STAGING 部署完成
  when: manual                      # ★ 手动触发
  only: [main]
  variables:
    DEPLOY_FILE: k8s/prod/deployment.yaml
    ENV: PROD
```

**安全设计**：
- DEV → STAGING → PROD 逐级串行，上一级完成后才能部署下一级
- PROD 须手动触发 (`when: manual`)，防止误操作
- PROD 限制 `only: main`，分支代码不能部署生产

### CI Variables 说明

| 变量 | 来源 | 说明 |
|------|------|------|
| `REGISTRY` | YAML 内置 | Harbor 地址 `192.168.1.61` |
| `IMG` | 计算 | 镜像全路径 `${REGISTRY}/tomcat-demo/tomcat-app` |
| `TAG` | CI 内置变量 | `v${CI_PIPELINE_IID}-${CI_COMMIT_SHORT_SHA}` 可追溯 |
| `HARBOR_USER` / `HARBOR_PASS` | GitLab CI Variable | Harbor 登录凭证 |
| `GITLAB_TOKEN` | GitLab CI Variable | GitLab API Token (oauth2 格式) |
| `DOCKER_BUILDKIT: "0"` | YAML 内置 | 禁用 BuildKit 以兼容旧版 Docker |
| `MAVEN_OPTS` | YAML 内置 | `.m2` 本地仓库路径，利于 CI cache |

### 为什么用 maven:3.9-eclipse-temurin-17-alpine？

- 兼容 Java 17，支持现代语法特性
- Alpine 轻量版，拉取速度快
- 与 `cache` 配置配合，缓存 `.m2` 依赖目录

## 二、Dockerfile（多阶段构建）

**文件位置**：`配置文件/tomcat-app/Dockerfile`

> 📄 **独立配置**: [配置文件/tomcat-app/Dockerfile](配置文件/tomcat-app/Dockerfile)

```dockerfile
# Stage 1: Maven Build
FROM 192.168.1.61/library/maven:3.8-openjdk-11-slim AS build
COPY pom.xml .
RUN mvn dependency:go-offline -B     # ★ 先下载依赖（缓存）
COPY src/ src/
RUN mvn clean package -DskipTests -B

# Stage 2: Tomcat Runtime
FROM 192.168.1.61/library/tomcat:9.0-jdk11-openjdk-slim
COPY --from=build /app/target/*.war /usr/local/tomcat/webapps/ROOT.war
```

### 设计决策

| 决策 | 原因 |
|------|------|
| **多阶段构建** | 最终镜像不含 Maven (~200MB)，仅含 Tomcat + WAR (~50MB) |
| **pom.xml 先于 src/ 复制** | 利用 Docker 层缓存，源码没变就不重新下载依赖 |
| **WAR → ROOT.war** | 部署为根路径 `/`，而非 `/tomcat-app/` |
| **FROM Harbor 地址** | 内网适配，避免 Docker Hub 拉取超时 |

## 三、GitLab 部署配置

**文件位置**：`配置文件/gitlab/gitlab-deploy.yaml`

> 📄 **独立配置**: [配置文件/gitlab/gitlab-deploy.yaml](配置文件/gitlab/gitlab-deploy.yaml)

### 3 个 PVC 分离存储

```yaml
gitlab-config  (5Gi)   # /etc/gitlab → GitLab 配置文件
gitlab-logs   (10Gi)   # /var/log/gitlab → 运行日志
gitlab-data   (30Gi)   # /var/opt/gitlab → Git 仓库数据 + PostgreSQL
```

### 为什么禁用监控组件？

```ruby
prometheus_monitoring['enable'] = false
alertmanager['enable'] = false
grafana['enable'] = false
```

GitLab 内置的 Prometheus/Grafana 需要额外 ~1GB 内存。POC 环境禁用可大幅降低资源消耗。

### Puma worker 优化

```ruby
puma['worker_processes'] = 1     # 默认 4 → 节省 ~1GB 内存
sidekiq['max_concurrency'] = 10  # 默认 50 → 节省 ~200MB 内存
```

### 健康检查超时配置

| 探针 | initialDelaySeconds | 原因 |
|------|---------------------|------|
| readinessProbe | 300 | GitLab 首次启动需初始化数据库 |
| livenessProbe | 360 | 给足够的启动时间，避免被 K8S 误杀 |

## 四、GitLab Runner 配置

**文件位置**：`配置文件/gitlab/gitlab-runner-deploy.yaml`

> 📄 **独立配置**: [配置文件/gitlab/gitlab-runner-deploy.yaml](配置文件/gitlab/gitlab-runner-deploy.yaml)

### 为什么用 Docker Executor + docker.sock？

| 方案 | 架构 | 优缺点 |
|------|------|--------|
| Docker Executor + 挂载 docker.sock | Runner → docker.sock → 宿主 Docker | ✅ 简单，⚠️ 安全风险（挂载 docker.sock） |
| Docker-in-Docker (DinD) | Runner → dind 容器 → 独立 Docker | ✅ 安全隔离，❌ 需要 privileged 模式 |

POC 环境选择第一种，简单直接。

### 自动注册机制

```bash
# initContainer 将注册脚本复制到共享卷
# 主容器启动时执行注册脚本，等待 GitLab 就绪后自动注册
gitlab-runner register \
  --executor "docker" \
  --docker-volumes "/var/run/docker.sock:/var/run/docker.sock"
```

## 六、K8S Deployment

**文件位置**：`配置文件/tomcat-app/deployment.yaml`

> 📄 **独立配置**: [配置文件/tomcat-app/deployment.yaml](配置文件/tomcat-app/deployment.yaml)

### GitOps 自动更新的行

```yaml
# ★ GitLab CI 的 sed 命令会精确替换这一行
image: 192.168.1.61/tomcat-demo/tomcat-app:v1-gitops-init
#          ↓ sed 替换后 ↓
# image: 192.168.1.61/tomcat-demo/tomcat-app:v5-a1b2c3d
```

### 为什么 replicas=2？

- 单副本：更新时有短暂不可用
- 双副本：滚动更新时始终保持至少一个 Pod 在线
- 对于演示应用足够，生产环境可调整

### imagePullSecrets 的必要性

```yaml
imagePullSecrets:
  - name: harbor-regcred
```

Harbor 项目 `tomcat-demo` 设为 public 可以不要，但 private 时必须配置。

## 七、CI/CD Pipeline 决策总结

| 决策 | 原因 |
|------|------|
| **9 Stage** | 质量门禁 + 多环境 = 完整生产级流水线 |
| **checkstyle + spotbugs** | 代码质量在编译前拦截，成本最低 |
| **YAML Anchor 模板复用** | 3 个 deploy job 共享同一套逻辑，DRY 原则 |
| **deploy 串行 (DEV→STAGING→PROD)** | 渐进式发布，生产部署前需经过完整验证 |
| **PROD 手动触发** | 最后一道安全防线 |
| **`[skip ci]`** | 防止部署提交触发新的 Pipeline |
| **`git checkout -B main`** | GitLab CI 默认 detached HEAD，必须绑定分支 |
| **先 rebase 后 sed** | 先同步远程变更，再修改文件，避免合并冲突 |
| **`gitlab.gitlab.svc.cluster.local`** | K8s 集群内 Pod 间通信使用 Service DNS |

## 八、方案 1 vs 方案 2 配置差异速查

| 配置项 | 方案 1 (Jenkins) | 方案 2 (GitLab) |
|--------|-----------------|-----------------|
| CI 文件 | `Jenkinsfile` (Inline Script XML) | `.gitlab-ci.yml` (YAML, 9 Stage) |
| 代码仓库协议 | `git://` (9418) | `http://` (80) |
| 部署清单位置 | `01-app/deployment.yaml` | `k8s/{dev,staging,prod}/deployment.yaml` |
| 清单如何更新 | Jenkins Stage 4 sed | GitLab CI deploy 模板 sed |
| 清单推送到哪 | 另一个仓库 (可观测性) | 同一个仓库 (tomcat-app) |
| 镜像标签格式 | `yyyyMMdd-HHmmss-BUILD` | `v{PIPELINE_ID}-{SHORT_SHA}` |
| Docker 镜像构建方式 | Stage 直接 docker build | Stage 2 独立 docker build |
| 基础镜像替换 | Jenkinsfile sed 改 Dockerfile | Dockerfile 直接写 Harbor 地址 |
