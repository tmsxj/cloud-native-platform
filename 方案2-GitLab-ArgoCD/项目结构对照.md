## 三、Git 仓库对照

### 方案 1: 双仓库模式

```
snownlp-observability-demo/  (Git Server)
├── 01-app/deployment.yaml  ← Jenkins sed 后 push 到这里
├── Jenkinsfile
└── src/

可观测性/           (Git Server)
└── argocd/application.yaml ← ArgoCD 从这里同步监控栈
```

### 方案 2: 单仓库模式

```
tomcat-app/                  (GitLab)
├── .gitlab-ci.yml           ← GitLab CI 流水线 (9 Stage)
├── k8s/
│   ├── dev/deployment.yaml  ← GitLab CI sed 后 push DEV 清单
│   ├── staging/deployment.yaml ← GitLab CI sed 后 push STAGING 清单
│   └── prod/deployment.yaml ← GitLab CI sed 后 push PROD 清单(手动触发)
└── src/
```

**关键区别**：方案 2 的代码和部署清单在**同一个仓库**，方案 1 需要 2 个仓库协作。

## 四、CI/CD 流程对照

### 方案 1 (Jenkins)

```
开发者 push 代码
  → Jenkins Poll SCM 检测变更（或手动触发）
    → git clone 到 Jenkins 工作空间
    → docker build + push
    → git clone 可观测性
    → sed 可观测性 的 deployment.yaml
    → git push 可观测性
      → ArgoCD 检测 可观测性 变更
        → kubectl apply 新 Deployment
```

### 方案 2 (GitLab CI)

```
开发者 push 代码
  → GitLab CI 自动触发 (检测 .gitlab-ci.yml)
    → GitLab Runner 拉取 Maven 镜像
    → unit-test (单元测试) + 覆盖率报告
    → checkstyle + spotbugs (质量门禁, 并行)
    → mvn package (编译)
    → docker build + push
    → deploy 模板: sed 仓库中 k8s/{dev,staging,prod}/deployment.yaml
    → git push 同一个仓库 (带 [skip ci])
      → ArgoCD 检测各环境清单变更
        → kubectl apply 各环境新 Deployment
```

### 优势对比

| 对比项 | 方案1 | 方案2 |
|--------|-------|-------|
| 仓库数量 | 2 个 | 1 个 |
| CI 触发 | 手动 / Poll SCM | 自动 (push 触发) |
| CI 配置格式 | XML + Groovy | YAML (9 Stage) |
| 代码与清单关系 | 分离在不同仓库 | 同仓库 |
| 质量门禁 | 无 | unit-test + checkstyle + spotbugs |
| 多环境部署 | 单一环境 | DEV → STAGING → PROD 渐进式 |
| 依赖组件 | Jenkins + Git Server | GitLab (自带 CI) |

## 七、从方案 1 迁移到方案 2 的关键区别

如果你是先做方案 1 再做方案 2，注意以下几点：

1. **GitLab 资源大**: GitLab CE 需要 2-4GB 内存，确保集群资源充足
2. **Runner 注册**: 与 Git Server (git-daemon) 不同，需要手动注册 Runner
3. **认证方式不同**: GitLab 用 HTTP + Token/JWT，不像 git-daemon 无认证
4. **CI 环境变量**: GitLab CI Variables 替代 Jenkins Credentials
5. **流水线定义**: YAML 替代 Groovy，语法完全不同
6. **ArgoCD 仓库注册**: http:// 协议替代 git:// 协议
