# 故障排查手册

> 基于 GitLab + ArgoCD GitOps 方案的实战踩坑记录。
> 与方案 1（Jenkins + Git Server）的故障不同，GitLab 的问题更多集中在资源、Runner 认证和 YAML 语法。
>
> 📁 **独立配置文件**: 文中涉及的完整 YAML/XML 在 [配置文件/](配置文件/) 目录。

## 一、GitLab 部署问题

### 1.1 GitLab Pod 启动超时被 K8S 杀死

**现象**：
```
Pod gitlab-xxx 反复重启，describe 显示 Liveness probe failed
```

**根因**：GitLab 首次启动需要初始化数据库、编译配置，耗时远超默认探针的等待时间。

**排查**：
```bash
# 查看 Pod 状态和事件
kubectl describe pod -n gitlab -l app=gitlab

# 实时查看日志
kubectl logs -n gitlab -l app=gitlab -f
```

**修复**：
```yaml
# 增大 initialDelaySeconds
readinessProbe:
  initialDelaySeconds: 300  # 3→5 分钟
livenessProbe:
  initialDelaySeconds: 360  # 3→6 分钟
  failureThreshold: 10      # 默认3→10
```

---

### 1.2 GitLab 初始 root 密码获取

**现象**：不知道 root 密码，无法登录 GitLab

**获取密码**：
```bash
# 方法1: 查看初始化文件 (24小时内有效)
kubectl exec -n gitlab deployment/gitlab -- cat /etc/gitlab/initial_root_password

# 方法2: 进入容器重置
kubectl exec -n gitlab deployment/gitlab -it -- bash
gitlab-rake "gitlab:password:reset[root]"
# 输入新密码
```

---

### 1.3 GitLab 内存不足导致 502

**现象**：GitLab 页面返回 502 Bad Gateway

**根因**：GitLab CE 最低需要 2GB 内存，Puma worker 过多会 OOM。

**修复**：在 `GITLAB_OMNIBUS_CONFIG` 中减少 worker 数量：
```ruby
puma['worker_processes'] = 0   # 最省内存（单线程模式）
sidekiq['max_concurrency'] = 5
postgresql['shared_buffers'] = '128MB'
```

## 二、GitLab Runner 问题

### 2.1 Runner 注册失败 "x509: certificate signed by unknown authority"

**现象**：Runner 注册时报 SSL 证书错误

**根因**：Runner 通过 HTTPS 连接 GitLab 但使用了自签名证书。

**修复**：
```bash
# 方案A: Runner 注册时跳过 TLS 验证
gitlab-runner register \
  --tls-ca-file="" \
  --url "http://gitlab.gitlab.svc.cluster.local"  # 集群内用 HTTP

# 方案B: 集群内部通信用 HTTP 而非 HTTPS
# 我们的部署配置已使用 HTTP (http://gitlab.gitlab.svc.cluster.local)
```

---

### 2.2 CI Pipeline 卡在 "Pending" 状态

**现象**：GitLab CI 显示 "This job is stuck because you don't have any active runners"

**排查**：
```bash
# 检查 Runner 注册状态
kubectl logs -n gitlab deployment/gitlab-runner

# 在 GitLab Admin 界面查看 Runner 状态
# Admin Area → CI/CD → Runners
```

**修复**：
1. 确认 `REGISTRATION_TOKEN` 正确（每次 GitLab 重启可能变化）
2. 重新注册 Runner：

```bash
kubectl delete pod -n gitlab -l app=gitlab-runner
# Runner 重启后会自动重新注册
```

---

### 2.3 Docker 命令在 CI 中不可用

**现象**：CI Pipeline 执行 `docker build` 时报 `docker: not found`

**根因**：Runner 容器没有 docker CLI 或 docker.sock 未挂载。

**排查**：
```bash
# 进入 Runner Pod 测试
kubectl exec -n gitlab deployment/gitlab-runner -- docker version
kubectl exec -n gitlab deployment/gitlab-runner -- ls -la /var/run/docker.sock
```

**修复**：
- 确认 `gitlab-runner-deploy.yaml` 中 `docker-sock` volume 正确挂载
- 确认 CI Job 使用了正确的镜像 tag (`docker:24-cli`)

---

### 2.4 CI Push 回 GitLab 失败 "Authentication failed"

**现象**：
```
remote: HTTP Basic: Access denied
fatal: Authentication failed
```

**根因**：未配置 `GITLAB_TOKEN` CI Variable，或 Token 权限不足。

**修复**：
1. 在 GitLab 创建 Personal Access Token（最少 `write_repository` 权限）
2. 添加到项目 CI Variables：`Settings → CI/CD → Variables → GITLAB_TOKEN`

## 三、.gitlab-ci.yml 问题

### 3.1 Pipeline 不触发

**现象**：push 代码后 GitLab 不触发 Pipeline

**排查**：
```bash
# 确认 .gitlab-ci.yml 在仓库根目录
git ls-tree HEAD

# 验证 YAML 语法
# 在 GitLab UI: CI/CD → Pipelines → CI Lint
```

**常见原因**：
- `.gitlab-ci.yml` 不在仓库根目录
- YAML 语法错误（缩进用空格不是 Tab）
- 分支是 feature 分支但 Pipeline 只触发 main

---

### 3.2 Maven Build 失败 "Could not resolve dependencies"

**现象**：
```
[ERROR] Failed to execute goal on project tomcat-app:
Could not resolve dependencies
```

**根因**：Maven 镜像无法访问 Maven Central 下载依赖。

**修复**：
```bash
# 方案1: 在 pom.xml 中添加 Maven 仓库镜像
# 方案2: 使用已缓存的本地仓库卷
# 方案3 (推荐): pom.xml 中仅依赖 javax.servlet-api (scope=provided)
#    Tomcat 已自带，Maven 编译时也不需要下载额外的 JAR
```

我们的 `pom.xml` 已经只依赖 `javax.servlet-api`（provided scope），通常不会遇到此问题。

---

### 3.3 Sed 替换不生效

**现象**：Stage 4 Update GitOps 执行后，deployment.yaml 中的 image 行没变化

**排查**：
```bash
# 在 CI 日志中查看替换前后对比
grep 'image:' k8s/deployment.yaml
```

**原因**：sed 模式不匹配。确认 `IMAGE_NAME` 变量的值和 deployment.yaml 中实际的 image 前缀一致。

**验证**：
```bash
echo "image: 192.168.1.61/tomcat-demo/tomcat-app:v1-old" | \
  sed "s|image: 192.168.1.61/tomcat-demo/tomcat-app:.*|image: 192.168.1.61/tomcat-demo/tomcat-app:v2-new|g"
# 期望输出: image: 192.168.1.61/tomcat-demo/tomcat-app:v2-new
```

## 四、ArgoCD 问题

### 4.1 无法连接 GitLab 仓库

**现象**：ArgoCD Application 状态 Unknown / Connection Error

**排查**：
```bash
# 测试 ArgoCD 能否连接 GitLab
kubectl exec -n argocd deployment/argocd-repo-server -- sh -c \
  "wget -q -O - http://gitlab.gitlab.svc.cluster.local/root/tomcat-app.git/info/refs"
```

**修复**：
- 确认 GitLab Service 可达：`kubectl get svc -n gitlab gitlab`
- 确认 ArgoCD 中仓库 URL 正确：
  ```bash
  argocd repo list --insecure --grpc-web
  ```

---

### 4.2 ArgoCD 一直 OutOfSync 但不部署

**现象**：ArgoCD UI 显示 OutOfSync 但始终不同步

**排查**：
```bash
# 查看 ArgoCD Application 事件
kubectl describe application tomcat-app -n argocd

# 手动触发同步
argocd app sync tomcat-app --insecure --grpc-web

# 查看同步日志
kubectl logs -n argocd deployment/argocd-application-controller | grep tomcat-app
```

**常见原因**：
- `k8s/` 目录路径不对（确认仓库中确实有该目录）
- `imagePullSecrets` 未配置，K8S 拉取镜像失败
- RBAC 权限不足

---

### 4.3 与方案1 ArgoCD 冲突

**现象**：两个 Application 状态正常但互相干扰

**说明**：方案1 和方案2 使用同一个 ArgoCD 实例，但目标是不同的命名空间 (`monitoring` vs `tomcat-demo`)，资源完全隔离，不会冲突。

```bash
# 确认两个 Application 各自独立
kubectl get application -n argocd
# 期望:
# NAME              SYNC STATUS   HEALTH STATUS
# snownlp-demo      Synced        Healthy
# tomcat-app        Synced        Healthy
```

**现象**：访问 Tomcat 应用返回 404

**排查**：

## 六、Pipeline 22 轮迭代实战经验（关键修复）

> 本次项目 CI/CD 流水线经过 **22 次 Pipeline 迭代**才达到稳定运行。以下是最关键的三个修复。

### 6.1 致命错误 1：detached HEAD 导致 git push 失败

**现象**：
```
fatal: You are not currently on a branch.
To push the history leading to the current (detached HEAD) state now, use
    git push origin HEAD:<name-of-remote-branch>
```

**根因**：GitLab CI 默认 checkout 代码时处于 **detached HEAD** 状态（基于特定 commit），不在任何分支上。直接 `git push` 会失败。

**排查**：
```bash
# 在 CI Job 中确认
git branch
# 输出: * (HEAD detached at xxxxxx)   ← 不在 main 分支上
```

**修复**：
```bash
git checkout -B main
# -B: 如果 main 已存在则重置，不存在则创建
# 从此之后的所有操作都在 main 分支上
```

### 6.2 致命错误 2：git rebase 与文件修改的顺序冲突

**现象**：
```
error: cannot rebase: You have unstaged changes.
error: cannot rebase: Your index contains uncommitted changes.
```

**根因**：如果你先执行 `sed` 修改文件，再执行 `git fetch + rebase`，Git 会拒绝在有未提交变更的工作区上 rebase。

**错误顺序**：
```bash
sed -i "s|image:.*|image: new-tag|" deployment.yaml  # 1. 先修改文件
git fetch origin main && git rebase origin/main         # 2. ❌ 失败了！有未提交的变更
```

**正确顺序**：
```bash
git checkout -B main
git fetch origin main && git rebase origin/main         # 1. ✅ 先同步远程
sed -i "s|image:.*|image: new-tag|" deployment.yaml     # 2. 再修改文件
git add deployment.yaml
git commit -m "[skip ci] deploy" && git push origin main
```

### 6.3 致命错误 3：K8s 集群内不能用 localhost 访问 GitLab

**现象**：
```
fatal: unable to access 'http://localhost/root/tomcat-app.git/':
Failed to connect to localhost port 80: Connection refused
```

**根因**：GitLab Runner Pod 和 GitLab Pod 都在 K8s 集群内，但 **不在同一个 Pod 内**。`localhost` 只能访问当前容器内部，不能跨 Pod 通信。

**排查**：
```bash
# 在 Runner Pod 内验证
kubectl exec -n gitlab deployment/gitlab-runner -- nslookup gitlab.gitlab.svc.cluster.local
# 应该解析到 GitLab Service 的 ClusterIP

kubectl exec -n gitlab deployment/gitlab-runner -- wget -qO- http://gitlab.gitlab.svc.cluster.local/-/health
# 应该返回 GitLab health check
```

**修复**：
```bash
# ❌ 错误写法
git remote set-url origin http://oauth2:TOKEN@localhost/root/tomcat-app.git

# ✅ 正确写法：使用 K8s Service DNS
git remote set-url origin http://oauth2:${GITLAB_TOKEN}@gitlab.gitlab.svc.cluster.local/root/tomcat-app.git
```

### 6.4 辅助修复：`[skip ci]` 防止循环触发

**现象**：deploy job 推送 commit 后，GitLab 又触发一轮新的 Pipeline，导致无限循环。

**根因**：deploy 阶段 push 的 commit 也会触发 Pipeline，新 Pipeline 再次 push，形成死循环。

**修复**：
```bash
# commit message 加 [skip ci] 或 [ci skip] 前缀
git commit -m "[skip ci] deploy DEV v58-abc1234"
# GitLab 识别到此标记后不会触发新的 Pipeline
```

## 七、快速诊断命令集

```bash
# ===== GitLab 诊断 =====
kubectl get pods -n gitlab -o wide
kubectl logs -n gitlab deployment/gitlab --tail=50
kubectl exec -n gitlab deployment/gitlab -- cat /etc/gitlab/initial_root_password

# ===== Runner 诊断 =====
kubectl get pods -n gitlab -l app=gitlab-runner
kubectl logs -n gitlab deployment/gitlab-runner

# ===== ArgoCD 诊断 =====
kubectl get application -n argocd
argocd app sync tomcat-app --insecure --grpc-web
argocd app history tomcat-app --insecure --grpc-web

# ===== Tomcat App 诊断 =====
kubectl get pods -n tomcat-demo -o wide
kubectl -n tomcat-demo get deploy tomcat-app \
  -o jsonpath='{.spec.template.spec.containers[0].image}'
kubectl logs -n tomcat-demo -l app=tomcat-app --tail=50

# ===== Harbor 诊断 =====
curl -u admin:Harbor12345 \
  http://192.168.1.61/api/v2.0/projects/tomcat-demo/repositories/tomcat-app/artifacts

# ===== 全链路状态一览 =====
echo "=== GitLab ===" && kubectl get pods -n gitlab
echo "=== ArgoCD ===" && kubectl get application -n argocd
echo "=== Tomcat ===" && kubectl get pods -n tomcat-demo
echo "=== Deployment Image ===" && \
  kubectl -n tomcat-demo get deploy tomcat-app \
    -o jsonpath='{.spec.template.spec.containers[0].image}{"\n"}'
```

## 八、常见问题速查表

| 症状 | 可能原因 | 解决方向 |
|------|---------|---------|
| GitLab 反复重启 | OOM / Liveness 超时 | 加大资源限制和探针延迟 |
| Runner 注册失败 | Token 错误 / 网络不通 | 检查 GitLab URL 和 Token |
| CI 卡在 Pending | 无活跃 Runner | 查看 Runner 状态 |
| docker: not found | docker.sock 未挂载 | 检查 Runner volume |
| Git push 401 | GITLAB_TOKEN 未配 / 权限不足 | 添加 CI Variable，确认 write_repository |
| Maven 依赖下载失败 | 无法访问 Maven Central | 检查 pom.xml 依赖 |
| detached HEAD push 失败 | GitLab CI 默认 detached HEAD | git checkout -B main |
| rebase 失败 (unstaged) | 先改文件后 rebase | 先 git fetch + rebase 再改文件 |
| localhost 连接失败 | Runner 跨 Pod 不能用 localhost | 改用 gitlab.gitlab.svc.cluster.local |
| Pipeline 循环触发 | deploy push 触发新 Pipeline | commit message 加 [skip ci] |
| ArgoCD Unknown | GitLab URL 不对 / 不可达 | 检查 repoURL |
| ImagePullBackOff | regcred 不存在 / 镜像不存在 | 重建 Secret 或先推镜像 |
| Tomcat 404 | WAR 包未部署 | 检查 webapps/ 目录 |
