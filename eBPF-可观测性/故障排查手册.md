# Cilium（基于 eBPF 的 CNI/网络方案） 故障排查手册

> 配套 `README.md`（部署实验全记录）/`Cilium-生产配置指南.md`（生产配置）
> 本手册提炼自真实部署踩坑，按「现象 → 原因 → 解决」组织，便于快速定位。

---

## 一、Agent CrashLoop：native 路由缺 CIDR

| 现象 | 排查 |
|:---|:---|
| Agent 反复重启，日志 `fatal: native routing cidr must be configured` | `routingMode: native` 要求节点间能直连 Pod（容器组） CIDR |

**解决**：K3s / 已有 CNI（容器网络接口） 环境改 `routingMode: tunnel`（VXLAN 隧道模式），并确认 `tunnelProtocol: vxlan`。

## 二、Agent CrashLoop：tunnel 与 autoDirectNodeRoutes 冲突

| 现象 | 排查 |
|:---|:---|
| 日志 `fatal: auto-direct-node-routes cannot be used with tunneling` | tunnel 模式下不能同时启用 `autoDirectNodeRoutes` |

**解决**：删除 `autoDirectNodeRoutes: true`，仅保留 `routingMode: tunnel`。

## 三、所有节点被 taint，CoreDNS Pending（agent-not-ready）

| 现象 | 排查 |
|:---|:---|
| 节点被打 `node.cilium.io/agent-not-ready:NoSchedule`，CoreDNS 等 Pod 全部 Pending | Agent 启动/重启期间打 taint；若 Agent 长期 CrashLoop，taint 永不被清除 |

**解决**：
1. 卸载重装：`helm uninstall cilium -n kube-system`
2. 手动清 taint：`kubectl taint nodes <node> node.cilium.io/agent-not-ready:NoSchedule-`
3. values 加 `agentNotReadyTaintKey: ""` 禁用该 taint（本项目已采用）

**根治**：在 `tolerations` 中追加 `node.cilium.io/agent-not-ready:NoSchedule` 容忍项，使滚动更新时即便残留 taint 也能调度，避免死锁。

## 四、Hubble Relay 无法连接 Agent gRPC server

| 现象 | 排查 |
|:---|:---|
| `hubble-relay` 重启，日志 `dial tcp ...:4244: connection refused` | `cni.enabled=false` 模式下 Agent 默认不启动 Hubble gRPC server（端口 4244） |

**解决**：values 显式设置 `hubble.listenAddress: ":4244"`，并 `hubble.enabled: true`。实测修正后 `cilium hubble status` 显示 `Hubble: Ok`，Relay / UI 稳定。

## 五、滚动更新时 agent-not-ready taint 死锁（Revision 5）

| 现象 | 排查 |
|:---|:---|
| `helm upgrade` 后部分节点 Cilium Pod 长时间 Pending，describe 报 `had untolerated taint {node.cilium.io/agent-not-ready}` | 旧 Pod 删除后节点残留 taint；新 Pod 的 tolerations 未包含该 taint → 不调度 → Agent 不 ready → taint 不清除，形成死锁 |

**解决**：
```bash
# 1. 清所有节点该 taint，让 Pod 能调度
kubectl taint nodes --all node.cilium.io/agent-not-ready:NoSchedule-
# 2. 个别节点仍 Pending，删 Pod 触发 DaemonSet 立即重建
kubectl delete pod <cilium-pod> -n kube-system
# 3. 验证全量就绪：DESIRED=READY=节点数，taint 仅剩 control-plane
kubectl get ds cilium -n kube-system
```
**根治**：values `tolerations` 追加 `node.cilium.io/agent-not-ready:NoSchedule`（本项目已启用，见 `cilium-values.yaml`）。

## 六、Grafana（可视化面板） 面板全空：数据源选错（坑 7）

| 现象 | 排查 |
|:---|:---|
| dashboard 导入后所有面板为空，`count(up{job="cilium-agent"})` 返回空 | 集群有两个 Prometheus（指标监控系统）：默认 `prometheus-server` 的 `serviceMonitorSelector` 不匹配 `prometheus: managed` 标签，不抓 Cilium；Cilium ServiceMonitor 精确命中 `prometheus-managed-0`（`prometheus-operated.monitoring:9090`） |

**解决**：Grafana 新建专用数据源指向 managed Prometheus：
- 名称 `Prometheus-Cilium`，uid `afrhyqopp15ogc`，类型 `prometheus`
- url `http://prometheus-operated.monitoring:9090`，access `proxy`
- dashboard 全部面板 / templating 绑定该 uid

**验证**：`count(up{job="cilium-agent"})=5`；agent 指标 `HAVE_DATA=18/18`。

**指标名提醒**：agent 默认已暴露 `cilium_drop_count_total` / `cilium_endpoint` / `cilium_endpoint_state` / `cilium_policy_endpoint_enforcement_status` 等，无需用 `+` 启用；`+cilium_endpoint_count`、`+cilium_datapath_drop_count_total` 等名称在 agent 中并不存在，会被静默忽略。

## 七、Hubble flow/drop/DNS 无数据（坑 8）

| 现象 | 排查 |
|:---|:---|
| Hubble 组件都起来了，但 Grafana 看不到 flow/drop/DNS 面板 | chart 只为 `cilium-agent` 建了 ServiceMonitor（9962），**没有**为 `hubble-metrics`（9965）建 ServiceMonitor，Hubble 指标未进 Prometheus |

**解决**：新增 `hubble-servicemonitor.yaml`，带 `prometheus: managed` 标签，selector 匹配 `hubble-metrics` Service（服务，集群内服务发现） 的 `k8s-app: hubble`，抓取 9965。

**验证**：`hubble_flows_processed_total`=44 series、`hubble_drop_total`=5、`hubble_tcp_flags_total`=20、`hubble_lost_events_total`=15。

**注意**：`hubble_dns_queries_total` 在安静 lab 环境为 0 series 属正常（需实际 DNS 流量经过被观测路径）；`hubble.metrics.enabled` 须写规范 `dns`（非 `dns:query`）。

## 八、Pod 跨节点网络不通（通用）

| 可能原因 | 排查 |
|:---|:---|
| 路由模式/隧道与底层网络不匹配 | `kubectl exec -n kube-system cilium-<pod> -- cilium status` 看 `Routing: Network: Tunnel [vxlan]` |
| kubeProxyReplacement 与 kube-proxy 冲突 | 二者互斥，只留其一；本项目 `kubeProxyReplacement: false`，Service NAT 由 kube-proxy 负责 |
| 连接追踪异常 | `cilium bpf ct list global` 看活跃连接；`cilium connectivity test` 做端到端校验 |
| IPAM 地址耗尽 | `cilium status` 看 IPAM allocated/total；本项目 `4/254 from 10.0.4.0/24` |

## 九、Hubble 无流量图 / UI 打不开

| 可能原因 | 排查 |
|:---|:---|
| Relay 未就绪 | `kubectl -n kube-system get pods -l k8s-app=hubble-relay` |
| Agent gRPC server 未起 | 确认 `hubble.listenAddress: ":4244"` 已设（坑 4） |
| UI Service 未暴露 | `kubectl -n kube-system get svc hubble-ui`；端口转发 `kubectl port-forward -n kube-system svc/hubble-ui 12000:80` |

## 十、快速诊断命令速查

```bash
# Agent / 集群状态
kubectl -n kube-system exec ds/cilium -- cilium status --verbose
kubectl -n kube-system exec ds/cilium -- cilium hubble status
kubectl -n kube-system exec ds/cilium -- cilium connectivity test

# eBPF / 连接追踪
kubectl -n kube-system exec ds/cilium -- cilium bpf lb list
kubectl -n kube-system exec ds/cilium -- cilium bpf ct list global

# Endpoint / 策略
kubectl get ciliumendpoints -A
cilium status | grep -i policy

# 节点 taint / DaemonSet
kubectl get nodes -o custom-columns=NAME:.metadata.name,TAINTS:.spec.taints
kubectl get ds cilium -n kube-system

# 卸载回滚（保留 Calico 回退备份）
helm uninstall cilium -n kube-system --wait
kubectl taint nodes node.cilium.io/agent-not-ready:NoSchedule- --all
kubectl get crds | grep cilium.io | awk '{print $1}' | xargs kubectl delete crd
```
