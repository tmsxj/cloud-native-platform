## 一、Jenkinsfile（CI 流水线）

**文件位置**：`snownlp-observability-demo/Jenkinsfile`

> 📄 **独立配置**: [配置文件/jenkins/jenkins-job-config.xml](配置文件/jenkins/jenkins-job-config.xml) — 完整的 Pipeline（流水线） Job XML 定义，带详尽中文注释。

**核心设计**：流水线使用 **Inline Script**（`CpsFlowDefinition`），而非 SCM 拉取外部 Jenkinsfile。这样可以避免 Git 协议兼容性问题（内网 Git Daemon 不支持 checkout scm）。

### 环境变量设计

```groovy
environment {
    HARBOR_REGISTRY = '192.168.1.61'
    IMAGE_NAME      = "${HARBOR_REGISTRY}/monitoring/snownlp-demo"
    IMAGE_TAG       = "${new Date().format('yyyyMMdd-HHmmss')}-${BUILD_NUMBER}"
}
```

| 变量 | 示例值 | 说明 |
|------|--------|------|
| `HARBOR_REGISTRY` | `192.168.1.61` | 内网 Harbor（私有镜像仓库） 地址 |
| `IMAGE_NAME` | `192.168.1.61/monitoring/snownlp-demo` | 镜像完整路径 |
| `IMAGE_TAG` | `20260625-101050-2` | 时间戳+构建号，确保唯一 |

> ⚠️ 注意：不能使用 `${currentBuild.time.format()}`，该语法只在 SCM-based Jenkinsfile 中有效。Inline Script 需要用 Groovy 的 `new Date().format()`。

### Stage 4 关键操作

```groovy
// deployment.yaml 中的 image 行：
// image: 192.168.1.61/monitoring/snownlp-demo:sw-agent
//                                          ↑
// sed 替换为 → image: 192.168.1.61/monitoring/snownlp-demo:20260625-101050-2

sed -i "s|image: ${IMAGE_NAME}:.*|image: ${IMAGE_NAME}:${IMAGE_TAG}|g" 01-app/deployment.yaml
```

**为什么用 sed**：Jenkins 不直接操作 K8s，而是修改 Git 中的 YAML 文件。Git 变更触发 ArgoCD（GitOps 持续交付工具） 同步，保证 Git 始终是 K8s 集群的真相源。

### Stage 5 Git Push

```groovy
git push origin HEAD:main
```

内网 Git Server 始终可达，push 失败即为真实错误，Pipeline 应立即报红而不是静默跳过。避免流水线变绿但 ArgoCD 无变更可同步的隐蔽故障。

## 二、ArgoCD Application — snownlp-demo

**文件位置**：`snownlp-observability-demo/argocd/snownlp-demo-app.yaml`

> 📄 **配置文件**: [配置文件/argocd/repo-secret.yaml](配置文件/argocd/repo-secret.yaml) — Git 仓库注册 Secret（密钥对象），ArgoCD 使用此凭证连接 Git Server。

### 完整配置

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: snownlp-demo
  namespace: argocd
spec:
  project: default
  destination:
    server: https://kubernetes.default.svc
    namespace: monitoring
  source:
    repoURL: https://your-git-server/snownlp-observability-demo.git
    targetRevision: main
    path: 01-app                         # 只监听业务应用目录
    directory:
      recurse: false
  syncPolicy:
    automated:
      prune: true                        # 自动删除 Git 中移除的资源
      selfHeal: true                     # 自动修复手动修改
      allowEmpty: false
    syncOptions:
      - CreateNamespace=true
      - PruneLast=true                   # 先部署新资源再删旧资源（零停机）
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2                       # 5s → 10s → 20s → 40s
        maxDuration: 3m
```

### 关键设计决策

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `path: 01-app` | 只监听业务目录 | 不扫描 promtail/loki/grafana 等配置文件 |
| `prune: true` | 自动清理 | 从 Git 删除文件时 K8s（Kubernetes，容器编排引擎） 资源也删除 |
| `selfHeal: true` | 自愈修复 | 手动 `kubectl edit` 会被回滚 |
| `PruneLast` | 零停机 | 先创建新 Pod（容器组） 再删除旧 Pod |

## 三、ArgoCD Application — monitoring-stack

**文件位置**：`可观测性/argocd/application.yaml`

### 与 snownlp-demo 的差异

| 配置项 | snownlp-demo | monitoring-stack | 原因 |
|--------|-------------|-----------------|------|
| `path` | `01-app/` | `.`（根目录） | 监控配置分散在根目录 |
| `recurse` | `false` | `true` | 监控配置有子目录 |
| `prune` | `true` | `false` | **监控基础设施不能自动删除** |
| `exclude` | 无 | `{backup-*,fixes,*.sh,*.md}` | 排除快照、脚本、文档 |
| `retry.limit` | 5 | 3 | 业务应用可能网络波动导致失败，多试几次 |

### 排除规则详解

```yaml
directory:
  recurse: true
  exclude: '{backup-*,fixes,*.sh,*.md,webhook-examples.md}'
```

- `backup-*`：备份快照目录（非 K8s 资源）
- `fixes/`：补丁 YAML（需手动 apply）
- `*.sh`：Shell 脚本（非 K8s 资源）
- `*.md`：文档文件（非 K8s 资源）

## 四、Dockerfile

**文件位置**：`snownlp-observability-demo/01-app/Dockerfile`

### 内网离线环境适配

```dockerfile
# 原始（需要 Docker Hub）：
FROM python:3.9-slim

# 修改后（内网 Harbor）：
FROM 192.168.1.61/library/python:3.11-slim
```

### 关键点：sw-python 启动方式

```dockerfile
ENTRYPOINT ["sw-python", "run"]
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
```

> **必须使用 `sw-python run` 而不是 `python app.py`！**
> `sw-python` 是 SkyWalking（APM 调用链追踪） Python Agent 的包装器。用 `python` 直接启动不会注入探针，SkyWalking UI 中将看不到调用链。

### 依赖安装顺序

```dockerfile
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt \
    && pip install --no-cache-dir apache-skywalking
```

分两步安装：`fastapi` 等基础依赖先装，`apache-skywalking` 后装。SkyWalking 的 `fastapi` 插件依赖 `fastapi` 包已存在。

### OCI 标签（由 Jenkinsfile 添加）

```bash
docker build \
    --label "git.commit=${GIT_COMMIT}" \
    --label "build.number=${BUILD_NUMBER}" \
    --label "build.time=${BUILD_TIMESTAMP}" \
    .
```

这些标签在镜像层中可追溯，便于关联到 Jenkins 构建记录。

## 五、K8s Deployment（部署，无状态工作负载）

**文件位置**：`snownlp-observability-demo/01-app/deployment.yaml`

### ⚡ GitOps（以 Git 为唯一真相源的运维模式） 标记

```yaml
# --- GitOps: 此标签由 Jenkins CI 自动更新, 请勿手动修改 ---
# Jenkins Pipeline: sed 替换 image: ...:任意tag → image: ...:构建号
# ArgoCD 检测到此行变化后自动同步到 K8s
image: 192.168.1.61/monitoring/snownlp-demo:sw-agent
```

`image:` 行是 GitOps 工作流中唯一会被 CI 自动修改的行。每次 CI 构建后，这个标签会被替换为新构建号。

### Prometheus（指标监控系统） 自动发现

```yaml
annotations:
  prometheus.io/scrape: "true"       # 启用抓取
  prometheus.io/port: "8000"         # 抓取端口
  prometheus.io/path: "/metrics"     # 抓取路径
```

Prometheus 的 `kubernetes_sd_config` 通过 Pod annotations 自动发现目标，无需手动配置 scrape target。

### SkyWalking 环境变量

```yaml
env:
  - name: SW_AGENT_NAME                          # 在 SkyWalking UI 中显示的服务名
    value: "snownlp-demo"
  - name: SW_AGENT_COLLECTOR_BACKEND_SERVICES    # OAP gRPC 地址（端口 11800）
    value: "skywalking-oap.monitoring:11800"
```

### ConfigMap（配置字典） 热更新机制

```yaml
volumes:
  - name: app-code
    configMap:
      name: snownlp-demo-app          # 包含 app.py 的 ConfigMap
containers:
  volumeMounts:
    - name: app-code
      mountPath: /app/app.py          # 覆盖镜像中的 app.py
      subPath: app.py                  # 只挂载一个 key
```

**优势**：修改代码只需更新 ConfigMap + 重启 Pod，不需要重新构建和推送 Docker 镜像（耗时 5-10 分钟），秒级生效。

### 健康检查

```yaml
livenessProbe:                         # 存活探针
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 30              # 等 30s 让镜像和 SkyWalking Agent 启动
readinessProbe:                        # 就绪探针
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 5               # 健康端点响应快，5s 即可
```

### Harbor 镜像拉取凭证

```yaml
imagePullSecrets:
  - name: harbor-regcred                # K8s Secret（dockerconfigjson 类型）
```

## 六、可观测性（三支柱关联）

### 支柱 1：Metrics（指标）

| 组件 | 配置位置 | 说明 |
|------|---------|------|
| app.py 暴露指标 | `app.py` 中 `/metrics` 端点 | `http_requests_total`(Counter) + `http_request_duration_seconds`(Histogram) |
| Prometheus scrape | `deployment.yaml` annotations | 自动发现 + 15s 抓取间隔 |
| 告警规则 | `可观测性/prometheus-alerting-rules.yml` | 15 条规则，8 组，三级路由 |

### 支柱 2：Logs（日志）

| 步骤 | 组件 | 配置 |
|------|------|------|
| 日志注入 trace_id | `app.py` SafeSWFilter | 每行日志自动附加 `[trace_id:xxx]` |
| 日志采集 | Promtail DaemonSet（守护进程集） | CRI 解析 → 提取 namespace/pod_name/trace_id |
| 日志存储 | Loki（日志系统） | `allow_structured_metadata: true` |
| 日志查询 | Grafana（可视化面板） Loki 数据源 | `derivedFields` 一键跳转 SkyWalking |

### 支柱 3：Traces（追踪）

| 组件 | 配置 | 说明 |
|------|------|------|
| Agent 启动 | `sw-python run` | 必须用此包装器 |
| gRPC 上报 | `skywalking-oap.monitoring:11800` | 通过 OAP 存储到 Elasticsearch |
| 手动埋点 | FastAPI 中间件 | 创建 Entry Span，标注 `Layer.Http` |

### 三支柱联动路径

```
用户请求 → app.py
    ├→ Traces: SkyWalking Agent 自动创建 Span → gRPC → OAP → ES
    ├→ Logs:   SafeSWFilter 注入 trace_id → stdout → Promtail → Loki
    └→ Metrics: /metrics Counter/Histogram → Prometheus scrape
                │
                ├→ AlertManager: 异常指标 → 邮件/Webhook 告警
                └→ Grafana: Loki 日志中 trace_id → derivedFields → SkyWalking URL
```

## 七、Jenkins Job 配置方式

> 📄 **独立配置**: [配置文件/jenkins/jenkins-deploy.yaml](配置文件/jenkins/jenkins-deploy.yaml) — Jenkins 全栈部署 YAML（PVC + SA + RBAC + Deployment + Service + Ingress（入口规则））。

### 为什么用 Inline Script 而非 SCM？

内网环境使用 Git Daemon（`git://` 协议），Jenkins 的 SCM-based Pipeline 对 Git Daemon 兼容性差。改用 Inline Script（`CpsFlowDefinition`）后：

1. Pipeline 脚本存储在 Jenkins 自身（`config.xml`）
2. 脚本中通过 `git branch: 'main', url: 'git://...'` 手动 clone
3. 避免了 Jenkins SCM 插件的协议限制

### Job 配置 XML 结构

```xml
<flow-definition plugin="workflow-job">
  <definition class="org.jenkinsci.plugins.workflow.cps.CpsFlowDefinition"
              plugin="workflow-cps">
    <script>pipeline { ... }</script>
    <sandbox>false</sandbox>
  </definition>
</flow-definition>
```

### 创建方式

```bash
# 通过 Jenkins REST API 创建
curl -X POST 'http://jenkins:8080/createItem?name=snownlp-demo-cicd' \
  -H 'Jenkins-Crumb: xxx' \
  -H 'Content-Type: application/xml' \
  --data-binary @config.xml
```

## 八、Jenkins 安全配置

当前集群使用 **Unsecured 模式**（无需登录），适合内网开发环境。

```xml
<!-- /var/jenkins_home/config.xml -->
<useSecurity>false</useSecurity>
```

如需启用安全：
```xml
<useSecurity>true</useSecurity>
<authorizationStrategy class="..."/>
<securityRealm class="hudson.security.HudsonPrivateSecurityRealm">
  <disableSignup>true</disableSignup>
</securityRealm>
```
