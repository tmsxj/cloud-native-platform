## 二、文件对照表

### 2.1 CI/CD 管线关键文件

| 文件 | 所属项目 | 作用 | 谁修改 |
|------|---------|------|--------|
| `Jenkinsfile` | snownlp-observability-demo | CI 流水线定义（6 Stage） | 开发者 |
| `argocd/snownlp-demo-app.yaml` | snownlp-observability-demo | ArgoCD 应用定义 | 运维 |
| `argocd/application.yaml` | 可观测性 | 监控栈 ArgoCD 应用定义 | 运维 |

### 2.2 业务应用文件（snownlp-observability-demo）

| 文件 | 修改者 | 修改频率 | 触发动作 |
|------|--------|---------|---------|
| `01-app/app.py` | 开发者 | 频繁 | 需重新构建镜像 |
| `01-app/Dockerfile` | 开发者 | 偶尔 | 需重新构建并推送基础镜像 |
| `01-app/deployment.yaml` | Jenkins CI | 每次构建 | ArgoCD 自动同步 |
| `01-app/service.yaml` | 开发者 | 极少 | ArgoCD 自动应用 |
| `01-app/loadgen.py` | 开发者 | 偶尔 | 需重新构建压测镜像 |
| `01-app/loadgen-deployment.yaml` | 开发者 | 偶尔 | ArgoCD 自动应用 |
| `01-app/requirements.txt` | 开发者 | 偶尔 | 需重新构建镜像 |

> **关键**：`deployment.yaml` 的 `image:` 行由 Jenkins 自动更新，开发者不直接修改。

### 2.3 可观测性配置（snownlp-observability-demo）

| 指标 | 采集方式 | 存储 | 可视化 | 告警 |

| **调用链** | SkyWalking Python Agent → OAP | Elasticsearch + PVC | SkyWalking UI | ❌ |

## 七、GitOps 仓库组织建议

```
Git 仓库根目录
├── argocd/                    ← ArgoCD Application 定义（推荐统一放置）
│   ├── snownlp-demo-app.yaml
│   └── application.yaml
├── 01-app/                    ← 被 ArgoCD 监听的应用目录
│   ├── deployment.yaml        ← ⚡ 自动更新的 K8s 清单
│   ├── service.yaml
│   └── Dockerfile
├── 02-promtail/               ← 可观测性组件（不被 ArgoCD 监听）
├── 03-loki/
├── 04-grafana/
├── 05-ingress/
├── Jenkinsfile                ← CI 流水线
└── README.md
```

**设计原则**：
1. **ArgoCD Application YAML 放在 `argocd/`**：统一管理，方便初始部署
2. **被监听的应用清单放在独立目录**：`path: 01-app/` 保证 ArgoCD 不扫描无关文件
3. **Dockerfile 和源码放在同一目录**：CI 构建时 `dir('01-app')` 直接工作
4. **部署清单和源码同仓库**：保持版本一致性，一个 tag 对应一组完整可部署的代码
