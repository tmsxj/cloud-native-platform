## 一、开场（30 秒）

## 二、项目架构（1 分钟）

### 先画张图（白板上或口头描述）

这个项目的观测链路分四条：

**第一条：请求链路**
```
用户请求 → Ingress → snownlp-demo 服务
    每个请求进来，SkyWalking Python Agent 在中间件层自动创建 Entry Span，
    标注 Layer=Http，然后通过 gRPC 上报到 OAP Server。
```

**第二条：日志链路**
```
应用 stdout 输出 → 容器运行时写 /var/log/pods/ → Promtail 采集
    → CRI 格式解析 → 正则提取 trace_id → 推送到 Loki 存储
```

**第三条：指标链路**
```
snownlp-demo 暴露 /metrics → Prometheus 自动抓取
    → Grafana 面板展示 QPS / 延迟 / 错误率
```

**第四条：关联跳转链路（核心亮点）**
```
Grafana Loki 面板看到一条日志：
  "2025-06-23 [INFO] trace_id=aa5e61... sentiment=NEGATIVE, text=客服态度恶劣"

     ↓ derivedFields 正则匹配 trace_id
     ↓ 构造 URL: skywalking.lab.local/trace?traceId=aa5e61...
     ↓ 前端渲染为可点击徽章
     ↓ 一键跳转到 SkyWalking 查看完整调用链
```

这就是从**发现问题 → 定位根因**的完整闭环。

## 三、技术选型和理由（1 分钟）

面试官可能会问"为什么选这套组合"，提前准备好：

| 组件 | 为什么选它 | 当前状态 |
|------|-----------|---------|
| **SkyWalking OAP + UI** | 原生 Python Agent，社区在国内活跃，自带 UI 比 Jaeger（链路追踪） 开箱即用 | ✅ 9.7.0，trace 数据存 ES |
| **Elasticsearch** | SkyWalking 的持久化存储后端（替代默认 H2），trace 数据可长期保留 | ✅ 8.11.0，给 OAP 用 |
| **Loki** | 日志存储——故意不存 ES。Loki 不建全文索引，按 label 检索，和 Grafana（可视化面板） 一套 UI，成本比 ELK 低得多 | ✅ 2.9.0 |
| **Promtail** | Loki 官方采集器，DaemonSet（守护进程集） 每节点一个，自动发现容器日志 | ✅ 5 节点全量覆盖 |
| **Prometheus** | 指标采集核心，自动发现带 annotation 的 Pod（容器组），含 node-exporter + kube-state-metrics | ✅ v3.11.3 |
| **Grafana** | 统一可视化，derivedFields 机制串联日志和追踪 | ✅ 12.3.1 |
| **AlertManager** | 15 条告警规则（8组），三级邮件路由（critical/warning/default），抑制规则防告警风暴 | ✅ 全链路打通 |
| **Ingress（入口规则） 单端口** | 家庭网络只映射一个 NodePort，Host 头路由分发四个域名 | ✅ |

### 关于为什么日志不存 ES 的选择

很多团队习惯 ELK 那套，日志直接灌 ES。我这个项目是故意选 Loki 的：
- ES 已经有 SkyWalking 的 trace 数据压力，不希望日志再跟 trace 抢 IO
- Loki 的 label 索引方式在"按 namespace/pod 查日志"的场景下比 ES 快
- Grafana 原生支持 Loki，derivedFields 一键跳转 SkyWalking 也不用额外开发

## 四、核心实现细节（2 分钟，面试重点）

### 4.1 trace_id 是怎么注入到日志里的？

这是我最核心的改动。Python 代码里写了一个 `SafeSWFilter`：

```python
class SafeSWFilter(logging.Filter):
    def filter(self, record):
        ctx = get_context()                    # 拿到 SkyWalking 当前上下文
        if ctx and ctx.segment:
            record.trace_id = str(ctx.segment.related_traces[0])  # 提取 traceId
        else:
            record.trace_id = "N/A"            # 安全降级
        return True
```

然后挂到 logger 上：

```python
logging.basicConfig(format='%(asctime)s trace_id=%(trace_id)s %(message)s')
logger.addFilter(SafeSWFilter())
```

这样每一行日志都自动带上 trace_id，不需要业务代码做任何改动。

### 4.2 Promtail 怎么提取 trace_id？

配置文件里是一个三阶段的 pipeline：

```yaml
pipeline_stages:
  - cri: {}                                    # 阶段1: 解析容器运行时格式
  - regex:                                     # 阶段2: 从日志正文中正则提取 trace_id
      expression: 'trace_id=(?P<trace_id>\w+)'
  - labels:                                    # 阶段3: 把 trace_id 存为 Loki label
      trace_id: ""
```

关键坑：**必须先 cri 再 regex**，因为日志先被 CRI 格式包裹了一层，如果顺序反了 regex 永远匹配不到。

### 4.3 Grafana 怎么实现一键跳转？

在 Loki 数据源的 JSON 配置里加 `derivedFields`：

```json
{
  "derivedFields": [
    {
      "matcherRegex": "trace_id=(\\w+)",
      "name": "SkyWalking Trace",
      "url": "http://skywalking.lab.local:31716/trace?traceId=${__value.raw}"
    }
  ]
}
```

Grafana 在渲染每条日志时，自动用正则匹配 `trace_id=xxx`，把匹配到的值填到 URL 模板里，前端渲染成可点击链接。

### 4.4 为什么手动创建 Entry Span？

FastAPI 的中间件运行在 SkyWalking Agent 的 ASGI 自动埋点之外。如果不手动创建，SkyWalking UI 里根本看不到这个 HTTP 端点。

```python
@app.middleware("http")
async def sw_trace_middleware(request, call_next):
    op = f"{request.method}:{request.url.path}"
    ctx = get_context()
    with ctx.new_entry_span(op=op) as span:
        span.layer = Layer.Http   # 标注为 HTTP 层
        response = await call_next(request)
        return response
```

`Layer.Http` 这个标注很重要——不标的话 SkyWalking 不知道这是什么类型的 Span，拓扑图里不会正确展示。

## 五、踩过的坑（1 分钟）

| 问题 | 现象 | 根因 | 怎么发现和解决的 |
|------|------|------|-----------------|
| 日志没 trace_id | Loki 日志全显示 trace_id=N/A | Agent 没正确初始化 | 排查后发现必须用 `sw-python run` 而不是 `python app.py` 启动，因为 agent 的自动注入依赖这个包装器 |
| Promtail 解析失败 | Loki 里查不到 pod_name label | CRI 日志有两层格式 | 看 Promtail 自身日志，发现 cargo 格式没解析，调 pipeline 顺序 |
| Grafana 查不到日志 | 刚部署时一片空白 | pod_name 含随机 UID | 把精确匹配改成正则模糊匹配 `pod_name=~"snownlp-demo.+"` |
| derivedFields 不生效 | 日志中没有可点击链接 | Loki 没开 `allow_structured_metadata` | 翻 Loki 文档才发现这个配置项，开启后 Promtail 提取的自定义 label 才能落地 |
| uvicorn 日志重复 | 每行日志输出两次 | uvicorn 默认 handler 和自定义 handler 并存 | 必须先 `uvicorn_logger.handlers.clear()` 再添加自己的 handler |

## 六、总结收尾（30 秒）

这个项目虽然看似只是一个 demo，但把可观测性的三个支柱完整覆盖了：**Traces、Logs、Metrics**，而且关键是把 **Logs 和 Traces 打通了**。

在真实业务场景里，运维看到一条异常日志，点一下就能看到完整的调用链，马上知道是哪个环节慢、哪个下游挂了。这就是可观测性的核心价值——缩短 MTTD（平均发现时间）和 MTTR（平均恢复时间）。

## 七、面试官可能的追问 & 准备思路

### Q: 你这个和直接用 ELK 有什么区别？
> ELK 的优势是全文检索，日志内容可以做复杂 grep。Loki 的优势是轻量和低成本——不需要建全文索引，按 label 查询，和 Grafana 统一可视化。而且我集群里 ES 已经在给 SkyWalking 存 trace 数据了，不想让日志再打一份进去抢 IO。两个场景分两条存储线，各用各擅长的方案。

### Q: 告警呢？你配了吗？
> 配了。15 条 Prometheus 告警规则，覆盖节点 CPU/内存/磁盘、Pod CrashLoop/NotReady、Deployment 副本数、PVC 容量、Prometheus 自身等 8 组。AlertManager 那边做了三级路由——critical 和 warning 走 QQ 邮箱 SMTP，还配了抑制规则防止告警风暴。飞书和企业微信的 webhook 方案也写好了，prometheus-webhook-dingtalk 做翻译层，接上就能用。

### Q: 如果有 1000 个微服务，你这个方案有什么问题？
> 首先是 SkyWalking OAP 的存储压力——单实例 ES 肯定扛不住，需要 ES 集群 + 合理保留策略。其次是 Promtail 的配置维护成本——1000 个服务意味着 1000 种日志格式，需要统一的配置管理和 pipeline 模板。还有就是 trace 采样率——全量 trace 存储成本太高，SkyWalking 的 sampling rate 得配。但核心架构是对的——Traces/Logs/Metrics 三条线分开，Loki 做日志、ES 做 trace 存储、Prometheus 做指标，不会互相拖累。

### Q: SkyWalking 和 Jaeger 你更推荐哪个？
> SkyWalking 功能更全——原生支持告警和指标，社区在中国更活跃。Jaeger 更轻量，和 OpenTelemetry（OTel，可观测性数据采集标准） 生态集成更好。如果团队主要是 Java/Go + 国内社区，选 SkyWalking；如果团队已经有 OpenTelemetry 的布局，选 Jaeger 更合适。

### Q: 日志和 trace 的关联，还有别的方案吗？
> 有的。另一个常见方案是 OpenTelemetry + 自定义 Exporter，直接把 Span ID 注入到日志的 MDC 里。还有一种是 Sidecar（边车代理） 模式，eBPF 层面的无侵入方案，但那个对业务代码的不可见性虽然好，灵活性就差一些。
