## 1. 这个 Demo 做了什么

### 1.1 一句话

> 一个 Python 情感分析微服务，三根数据线在一个应用里全部打通：**Trace（SkyWalking）→ Logs（Loki 含 trace_id）→ Metrics（Prometheus）→ Grafana 一键跳转**。

| `05-ingress/monitoring-ingress.yaml` | 69 行 | 4 域名 Host 路由 Ingress |

```

  ├─ grafana.lab.local    → Grafana

                    ┌──────────────────────┐
                    │    snownlp-demo       │
                    │    FastAPI :8000      │
                    │                       │
                    │  ┌─────────────────┐  │
                    │  │ sw_trace_mw ①  │──┼──→ gRPC → SkyWalking OAP → ES
                    │  │  Entry Span    │  │
                    │  │  Layer.Http    │  │
                    │  └─────────────────┘  │
                    │  ┌─────────────────┐  │
                    │  │ SafeSWFilter ② │──┼──→ stdout → CRI → Promtail → Loki
                    │  │  trace_id=xxx  │  │
                    │  └─────────────────┘  │
                    │  ┌─────────────────┐  │
                    │  │ /metrics ③     │──┼──→ Prometheus scrape
                    │  └─────────────────┘  │
                    └──────────────────────┘
```

### 2.2 一条请求的完整生命周期

```
loadgen POST /predict {"text":"客服态度恶劣"}
  │
  ├─ ① sw_trace_middleware
  │     → get_context() → ctx.new_entry_span(op="POST:/predict")
  │     → span.layer = Layer.Http
  │     → gRPC:11800 → SkyWalking OAP → Elasticsearch
  │
  ├─ ② SafeSWFilter.filter()
  │     → ctx.segment.related_traces[0] → trace_id=aa5e61...
  │     → logger.info("...trace_id=aa5e61... sentiment=NEGATIVE...")
  │     → stdout → /var/log/pods/.../0.log → Promtail → Loki
  │
  └─ ③ add_metrics middleware
        → REQUEST_COUNT.inc() / REQUEST_LATENCY.observe()
        → /metrics → Prometheus scrape → TSDB
```

### 2.3 关键关联：Grafana derivedFields

```
Grafana Explore → Loki 数据源
  日志行: "2026-06-23 trace_id=aa5e61... sentiment=NEGATIVE..."
       ↓ derivedFields matcherRegex 匹配 trace_id
       ↓ URL 构造: skywalking.lab.local:31716/trace?traceId=aa5e61...
       ↓ 前端渲染为可点击徽章
       → 一键跳转 SkyWalking 查看完整调用链
```

## 3. 操作步骤详解

### 3.1 代码编写

#### 3.1.1 `app.py` 核心模块分解

| 模块 | 行号 | 功能 |
|------|------|------|
| Prometheus 指标定义 | 32-41 | `http_requests_total` (Counter) + `http_request_duration_seconds` (Histogram) |
| SkyWalking 追踪中间件 | 59-68 | 手动创建 Entry Span，标注 `Layer.Http` |
| 指标采集中间件 | 75-90 | 记录每个请求的延迟和状态码 |
| /metrics 端点 | 98-100 | 暴露 Prometheus 抓取格式 |
| SafeSWFilter 日志过滤器 | 115-128 | 从 SW 上下文提取 trace_id 注入日志 |
| 日志系统配置 | 136-153 | `logging.basicConfig` + uvicorn handler 清理 |
| /predict 业务接口 | 163-190 | SnowNLP 情感分析 + 日志输出 |
| /health 健康检查 | 193-196 | K8s liveness/readiness probe |

#### 3.1.2 关键技术决策

**决策1：手动创建 Entry Span**

在 FastAPI 的 ASGI 中间件中不能用 SkyWalking 的自动埋点——Agent 的自动注入在 ASGI 协议层之上，中间件层面拦截不到。必须手动 `ctx.new_entry_span()`。

```python
with ctx.new_entry_span(op=op) as span:
    span.layer = Layer.Http  # 不标这行，拓扑图里看不到
    response = await call_next(request)
```

**决策2：日志过滤模式而非直接写 trace_id**

用 Python logging Filter 机制注入 trace_id，业务代码里一行不改：

```python
class SafeSWFilter(logging.Filter):
    def filter(self, record):
        record.trace_id = str(ctx.segment.related_traces[0])
        return True
# 业务代码只需:
logger.info(f"text='{text}', sentiment={s}")
# 输出自动变成:
# 2026-06-23 trace_id=aa5e61... text='...', sentiment=NEGATIVE
```

**决策3：ConfigMap 挂载代码**

`app.py` 通过 ConfigMap 的 subPath 挂载到 `/app/app.py`，覆盖镜像中的默认代码。修改代码 → 更新 ConfigMap → 重启 Pod，不用重建镜像。

### 3.4 Promtail 日志采集配置

**三阶段 pipeline 设计**：

```yaml
pipeline_stages:
  - cri: {}                          # ① 解析容器运行时包装
  - regex:                           # ② 从文件路径提取 K8s 元数据
      expression: '^/var/log/pods/(?P<namespace>[^_]+)_(?P<pod_name>[^_]+)_...'
      source: filename               # 从文件名提取，不受日志内容格式影响
  - labels:
      namespace:
      pod_name:
  - regex:                           # ③ 从日志内容提取 trace_id
      expression: 'trace_id=(?P<trace_id>\w+)'
  - labels:
      trace_id:                      # 提升为 Loki label → 供 Grafana derivedFields 使用
```

**更新命令**：

```bash
kubectl apply -f 02-promtail/promtail-config.yaml
kubectl rollout restart daemonset/promtail -n monitoring
```

## 4. 排障实录

### 4.1 trace_id 始终显示 N/A

**现象**：Grafana Loki 面板中所有日志行都显示 `trace_id=N/A`，无法关联到调用链。

**排查过程**：

1. `kubectl logs snownlp-demo-xxx` → 确认每行日志都有 `trace_id=N/A`
2. `kubectl exec snownlp-demo-xxx -- python -c "from skywalking.trace.context import get_context; print(get_context())"` → 返回 `None`
3. 确认代码逻辑 `SafeSWFilter.filter()` 逻辑正确，`try/except` 没有抛异常
4. 检查容器启动方式 → Deployment 的 `command` 字段

**根因**：容器直接用 `uvicorn app:app` 启动，而 SkyWalking Python Agent 的自动注入依赖 `sw-python run` 这个包装器——它负责在进程启动前初始化 Agent 探针。

**修复方案**：Dockerfile 和 Deployment 中改用：
```yaml
command: ["sw-python", "run"]
args: ["/usr/local/bin/uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
```

**教训**：
- `pip install apache-skywalking` ≠ Agent 已激活
- `import skywalking` 不报错 ≠ Agent 已初始化
- **必须用 `sw-python run` 启动进程**，这是 SkyWalking Python Agent 的硬性要求

### 4.2 sw-python 启动 CrashLoopBackOff

**现象**：改为 `sw-python run` 后，Pod 启动立即 Crash。

**排查过程**：

1. `kubectl logs snownlp-demo-xxx` → `ModuleNotFoundError: No module named 'skywalking'`
2. `kubectl exec snownlp-demo-xxx -- pip list` → `apache-skywalking` **未安装**
3. 检查 Docker build 日志 → 安装过程中报过 deprecation warning 但没有 error
4. 检查 `requirements.txt` 安装顺序

**根因**：Dockerfile 中 `pip install -r requirements.txt` 一次性安装所有包，但 `apache-skywalking` 的 pip 依赖解析需要在 `fastapi` 安装之后才能正常完成。如果 `apache-skywalking` 排在 requirements.txt 前面，且 pip 按顺序解析，会导致依赖关系未满足而静默跳过。

**修复方案**：Dockerfile 分两步安装：
```dockerfile
RUN pip install --no-cache-dir -r requirements.txt \
    && pip install --no-cache-dir apache-skywalking
```
先把 fastapi/uvicorn/snownlp/prometheus_client 装好，再单独装 skywalking。

**教训**：pip 在 Dockerfile 中的安装顺序不是"随机的"——某些 Agent 类库（SkyWalking、OpenTelemetry 等）有隐藏依赖链，分步安装是稳妥做法。

### 4.3 Promtail 提取不到 pod_name

**现象**：Loki 中日志正常进来，但 `pod_name` label 为空，按 pod 筛选日志失败。

**排查过程**：

1. `kubectl logs -n monitoring promtail-xxx` → Promtail 自身日志无报错
2. Grafana Explore 中查日志 → `{namespace="monitoring", pod_name!=""}` 返回空
3. 单步调试 pipeline → 发现 `regex` 提取阶段拿到的是原始 CRI 格式日志行，而非文件路径

**根因**：Promtail pipeline 的 `regex` 阶段的 `source: filename` 引用的是原始日志的元数据字段 `filename`。但如果在 `regex` 之前先执行了 `cri` 解析，`cri` 会改写日志行格式——然而 `filename` 这个元数据字段不会受影响（它是文件路径，不是日志内容的一部分）。

真正原因是：pipeline_stages 中 `cri` 必须在 `regex(labels)` 之前——但实际上 `source: filename` 的 regex 不会受 cri 影响。进一步排查发现是**日志文件路径格式与正则不匹配**——某些 Pod 的 UID 中含有额外字符。

**修复方案**：调整 CrI 日志采集路径正则，使其兼容更多路径变体。同时确认 `source: filename` 的语义——它永远是原始文件的绝对路径，不受任何 stage 修改影响。

**教训**：Promtail 的 `source: filename` 是最可靠的元数据来源，不受日志内容格式变化影响。排查 pipeline 问题时，先用最简单的 regex 测试 `source: filename`，确认路径正则无误后再加其他 stage。

### 4.4 Grafana 查不到日志

**现象**：Promtail + Loki 都跑着，但在 Grafana Explore 中查 `{namespace="monitoring"}` 无结果。

**排查过程**：

1. `curl http://loki.monitoring:3100/ready` → 返回 Ready
2. `kubectl logs promtail-xxx` → 正常推送日志到 Loki
3. Grafana 数据源连接测试 → OK
4. 尝试 `{namespace=~"monitoring"}` 精确匹配 → 返回空
5. 查 Loki label 值 → 发现 `pod_name` label 包含随机 UID 后缀，例如 `snownlp-demo-64c9555f89-8m92f`

**根因**：`pod_name` label 从文件路径提取，K8s 的 `/var/log/pods/` 路径中 pod 名称本身就包含 Deployment 生成的随机 UID。精确查询 `pod_name="snownlp-demo"` 匹配不到任何日志。

**修复方案**：查询时用正则模糊匹配：
```
{pod_name=~"snownlp-demo.+"}
```
而非：
```
{pod_name="snownlp-demo"}
```

**教训**：K8s 环境下的 `pod_name` 永远不等于 Deployment 名称——ReplicaSet 会追加随机 UID。日志查询永远用 `=~` 正则而非 `=` 精确匹配。

### 4.5 derivedFields 链接不出现

**现象**：Grafana Explore 中日志行正常显示，`trace_id=aa5e61...` 文本可见，但没有变成可点击链接。

**排查过程**：

1. Grafana Data Sources → Loki → derivedFields → 确认 JSON 配置已保存
2. Explore 中展开原始 JSON 日志 → `trace_id` label 存在
3. F12 浏览器开发者工具 → 无 JavaScript 报错
4. 检查 Grafana 版本 → 12.3.1 支持 derivedFields
5. 回查 Loki 配置

**根因**：Loki 配置中 `allow_structured_metadata: true` 未开启。默认情况下 Loki 不存储 Promtail 提取的自定义 label（如 `trace_id`）。Promtail 确实推送了 `trace_id`，但 Loki 端直接丢弃了。

**修复方案**：在 Loki ConfigMap 中开启：
```yaml
limits_config:
  allow_structured_metadata: true
```
然后 `kubectl rollout restart statefulset/loki`。重启后，**新写入的日志**才会携带 trace_id 元数据，旧日志不追溯。

**教训**：
- Promtail 推送 ≠ Loki 存储（中间有过滤层）
- `allow_structured_metadata` 是 Loki 2.9+ 的关键配置项，必须显式开启
- 开启后只对新数据生效，要等一会儿才有可点击链接

### 4.6 uvicorn 日志重复输出

**现象**：Loki 中同一条 HTTP 请求出现**两条**格式不同的日志：
- 一条带 `trace_id=xxx`（我们的自定义格式）
- 一条 `GET /health HTTP/1.1 200`（uvicorn 默认格式）

**排查过程**：

1. 确认两条日志来自同一个 Pod、同一个容器
2. 检查 uvicorn 日志配置代码 → 添加了自定义 handler 和 filter
3. 检查 uvicorn 源码 → uvicorn 启动时**自动注册**一个默认的 `logging.StreamHandler` 到 `uvicorn.access` logger

**根因**：Python logging 框架中，logger 可以挂多个 handler。uvicorn 启动时注册了一个默认 handler（输出朴素格式），我们的代码又 add 了一个自定义 handler（输出带 trace_id 的格式）。两个 handler 同时触发，同一条日志输出两次。

**修复方案**：注册自定义 handler 前先清掉旧的：
```python
uvicorn_logger = logging.getLogger("uvicorn.access")
uvicorn_logger.handlers.clear()  # ← 必须清！
handler = logging.StreamHandler()
handler.setFormatter(...)
uvicorn_logger.addHandler(handler)
uvicorn_logger.addFilter(SafeSWFilter())
```

**教训**：框架（uvicorn/Django/Gin）通常会在初始化时注册默认日志 handler。如果要在应用代码中自定义日志格式，**永远先 `.handlers.clear()`**，否则就是"追加"而非"替换"。

### 4.7 Docker 镜像构建后推送到私有仓库失败

**现象**：`docker push 192.168.1.61/monitoring/snownlp-demo:sw-agent` 报 `http: server gave HTTP response to HTTPS client`。

**排查过程**：

1. `docker login 192.168.1.61` → 成功
2. `docker push` → 报 HTTP/HTTPS 协议错
3. 检查 Docker daemon 配置 → `insecure-registries` 中未包含 Harbor 地址

**根因**：Harbor 私有仓库使用 HTTP 而非 HTTPS，Docker 默认要求所有 registry 必须是 HTTPS 连接。

**修复方案**：修改 Docker daemon 配置（`/etc/docker/daemon.json`）：
```json
{
  "insecure-registries": ["192.168.1.61:80"]
}
```
然后 `systemctl restart docker`。

**教训**：自建 Harbor/Registry 如果用 HTTP 协议，必须配置 `insecure-registries`。K8s 集群所有节点的 containerd/docker 都要配。

### 4.8 loadgen 压测无 trace 数据

**现象**：loadgen 正常发送请求到 snownlp-demo，但 SkyWalking UI 中看不到调用链。

**排查过程**：

1. `kubectl logs snownlp-demo-xxx` → 日志正常输出，`trace_id=xxx` 有值
2. SkyWalking UI 中 → Services 列表有 `snownlp-demo`，但无 trace 数据
3. `kubectl logs skywalking-oap-xxx` → gRPC 连接正常建立
4. 检查 OAP 配置 → ES 存储后端配置正确

**根因**：loadgen 使用的是 `urllib.request` 而非 `requests` 或 HTTP 客户端库，SkyWalking Python Agent 的自动埋点只拦截标准库中的 `urllib` 出站请求……实际上 gRPC 上报到 OAP 是通的，问题出在 OAP 的**刷写延迟**——OAP 默认每 25 秒批量刷写一次 trace 到 ES，UI 需要等。

**修复方案**：等待 30 秒后刷新 SkyWalking UI。

**教训**：SkyWalking OAP → ES 的刷写不是实时的，默认每 25s 一批。部署完不要立刻断言"没有 trace"，等半分钟再刷新。

### 4.9 SkyWalking UI trace 页面偶发空白

**现象**：SkyWalking UI 中进入 trace 详情页，偶尔页面空白。

**排查过程**：

1. F12 查看网络请求 → GraphQL 查询返回 `{"errors":[...]}` 
2. 刷新后有时正常 → 问题间歇性出现
3. 检查 SkyWalking OAP 日志 → 间歇性 `connect timeout` 到 ES

**根因**：单节点 ES 在 JVM 内存紧张时触发 GC，延迟过高导致 OAP 的查询超时。演示环境资源有限，1 个 ES 实例扛 OAP 的写入 + UI 的查询。

**修复方案**：演示场景可接受——刷新一下一般就好。生产环境需要 ES 集群 + 调大 JVM heap。

**教训**：单节点 ES 既是写入端也是查询端时，GC 延迟不可控。SkyWalking UI 的 GraphQL 查询对延迟敏感。

## 6. 面试讲速版

### 6.1 三句话讲清楚

> "我写了一个 Python 情感分析微服务做可观测性演示。第一个亮点是**手动埋点**：在 FastAPI 中间件里手动创建 SkyWalking Entry Span，标注 Layer=Http，这是 Agent 自动埋点做不到的——FastAPI 的 ASGI 中间件跑在 Agent 拦截层之外。
>
> 第二个亮点是**日志和 trace 的关联**：写了一个 Python logging Filter，从 SkyWalking 上下文中提取 trace_id 注入到每一行日志，然后通过 Promtail 正则提取、Loki 结构化元数据存储、Grafana derivedFields 正则匹配，在 UI 中渲染成可点击链接——运维看到一条错误日志，点一下就能看到完整调用链。
>
> 第三个亮点是**三根数据线在一个应用里全通了**：/metrics 端点给 Prometheus、stdout 日志给 Promtail→Loki、gRPC trace 上报给 SkyWalking→ES，不是分开演示的，是真的一个请求三条线同时产生数据。"

### 6.2 面试官追问 & 拆解

| 如果问 | 可以答 |
|--------|--------|
| 为什么手动埋点？ | "FastAPI 的中间件运行在 SkyWalking Agent 的 ASGI 自动埋点层之外，不手动创建 Entry Span 的话 SkyWalking UI 根本看不到这个端点。" |
| 为什么不是 ELK？ | "ES 已经被 SkyWalking 占用了——trace 数据需要聚合查询，ES 的倒排索引正合适。日志只需要按 namespace/pod + 时间翻页，Loki 的 label 索引模式更轻量。两条存储线互不影响。" |
