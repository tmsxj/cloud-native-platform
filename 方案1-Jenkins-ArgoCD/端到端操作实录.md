# 端到端操作实录：从零搭建 Jenkins + ArgoCD GitOps 全链路

> 本文档记录了从空 K8s 集群开始，到代码 push → Jenkins 构建 → ArgoCD 同步全过程的所有命令。
> 适用于在离线/内网环境中复现整套系统。
>
> **零占位符，零省略**：每个命令都可以直接复制执行（替换环境变量后即可）。
>
> 📁 **独立配置文件目录**: [配置文件/](配置文件/) — 所有 YAML/XML/Shell 配置已提取为独立文件，带详尽中文注释，可直接复制使用，避免文档格式混乱。

## 〇、前置准备（必读）

以下环境变量值贯穿全文，**你需要根据自己的环境修改后，全文替换**：

| 变量 | 本文档值 | 说明 |
|------|---------|------|
| `HARBOR_IP` | `192.168.1.61` | Harbor 镜像仓库地址 |
| `HARBOR_USER` | `admin` | Harbor 登录用户名 |
| `HARBOR_PASS` | `Harbor12345` | Harbor 登录密码 |
| `K8S_MASTER_IP` | （集群主节点 IP） | 用于 DNS/hosts 映射 |

以下域名需要在你的 **本地机器**（kubectl 执行端）配置 `/etc/hosts`（Windows: `C:\Windows\System32\drivers\etc\hosts`）：

```
<K8S_MASTER_IP>  jenkins.test  argocd.test  grafana.lab.local  snownlp.lab.local
```

如果你用的是 NodePort 方式暴露服务而非 Ingress，则不需要 hosts，直接使用 `<K8S_MASTER_IP>:<NodePort>` 访问即可，后文中的域名替换为对应 IP+端口。

**操作机环境要求**：
- `kubectl` 已配置好集群访问
- `docker` CLI 可用（用于构建镜像）
- `argocd` CLI 已安装（[下载](https://github.com/argoproj/argo-cd/releases)）
- `git` CLI 已安装
- 集群已有 StorageClass（本文档用 `local-path`）
- 集群已有 Ingress Controller（本文档用 `nginx`）

**ArgoCD 初始密码**：每次安装都不一样，见 5.1 节获取命令，下文 `wRRzfFrgasxcpwwq` 仅为示例。

### 1.4 集群访问方式说明

本文档中所有对外访问 URL 使用 `jenkins.test:31716`、`argocd.test:31716` 等格式。这依赖：

```
Ingress Controller (nginx) NodePort 31716
  └─ Ingress jenkins     (host: jenkins.test)
  └─ Ingress argocd      (host: argocd.test)
  └─ Ingress grafana     (host: grafana.lab.local)
  └─ Ingress snownlp     (host: snownlp.lab.local)
```

**验证 Ingress Controller 端口**：
```bash
kubectl get svc -A | grep ingress
# ingress-nginx   ingress-nginx-controller   NodePort   10.x.x.x   <none>   80:31716/TCP,443:31717/TCP
```

**如果你的 Ingress Controller 用不同端口**，把下文所有 `:31716` 替换即可。

**如果没有 Ingress Controller**，改用 NodePort 直接暴露服务：
```bash
kubectl patch svc jenkins -n jenkins -p '{"spec":{"type":"NodePort"}}'
kubectl patch svc argocd-server -n argocd -p '{"spec":{"type":"NodePort"}}'
# 然后获取 NodePort 端口
kubectl get svc jenkins -n jenkins -o jsonpath='{.spec.ports[0].nodePort}'
kubectl get svc argocd-server -n argocd -o jsonpath='{.spec.ports[0].nodePort}'
```

### 2.4 ⚡ 关键踩坑：Git Server HEAD 引用

```bash
# 问题：git push 后 clone 出来是空目录
# 根因：bare repo 的 HEAD 指向了不存在的 refs/heads/master

# 检查 HEAD 指向

  "cat /git/repos/snownlp-observability-demo.git/HEAD"
# 输出: ref: refs/heads/master  ← 错误！

# 修复：修正为 main
kubectl exec -n git deployment/git-server -- sh -c \
  "cd /git/repos/snownlp-observability-demo.git && git symbolic-ref HEAD refs/heads/main"
```

### 3.1 基础镜像准备

```bash
# Harbor 地址: 192.168.1.61（独立服务器或 NodePort 暴露）
# 登录 Harbor
```

### 4.3 创建 Jenkins Pipeline Job（Inline Script 方式）

> 📄 **配置文件**: [配置文件/jenkins/jenkins-job-config.xml](配置文件/jenkins/jenkins-job-config.xml)
>
> 完整的 Pipeline Job XML 定义（6 个 Stage，带详尽中文注释）。
>
> **关键决策**：不用 SCM-based Pipeline（内网 Git Daemon 兼容性差），改用 Inline Script（CpsFlowDefinition）。

```bash
# ⚡ 通过 Jenkins REST API 创建 Job（推荐方式，避免手动写 config.xml 到磁盘导致 404）

# 步骤1: 获取 Jenkins Crumb Token（CSRF 保护）
CRUMB=$(curl -s 'http://jenkins.test:31716/crumbIssuer/api/json' | grep -o '"crumb":"[^"]*"' | cut -d'"' -f4)
COOKIE=$(curl -s -I 'http://jenkins.test:31716/crumbIssuer/api/json' | grep -i 'set-cookie' | cut -d: -f2- | tr -d '\r')

# 步骤2: 通过 REST API 创建 Job（使用配置文件目录中的 jenkins-job-config.xml）
curl -X POST 'http://jenkins.test:31716/createItem?name=snownlp-demo-cicd' \
  -H "Jenkins-Crumb: ${CRUMB}" \
  -H "Cookie: ${COOKIE}" \
  -H 'Content-Type: application/xml' \
  --data-binary @配置文件/jenkins/jenkins-job-config.xml

# 步骤3: 验证 Job 已创建
curl -s 'http://jenkins.test:31716/job/snownlp-demo-cicd/api/json' | grep -E 'name|buildable'
```

### 4.4 ⚡ 关键踩坑：Jenkins Job 404

```bash
# 问题：直接写 config.xml 到 Jenkins jobs 目录后 reload，Job 消失
# 根因：XML 格式/编码不兼容，Jenkins 解析失败就跳过

# ❌ 错误做法：
#   echo "config.xml content" > /var/jenkins_home/jobs/snownlp-demo-cicd/config.xml
#   curl -X POST http://jenkins:8080/reload

# ✅ 正确做法：永远通过 REST API 创建/更新 Job
#   POST /createItem?name=xxx        → 创建新 Job
#   POST /job/xxx/config.xml         → 更新已有 Job 配置

# 如果已经出错，修复步骤：
kubectl exec -n jenkins deployment/jenkins -- sh -c \
  "rm -rf /var/jenkins_home/jobs/snownlp-demo-cicd"

# 然后重新用 API 创建（见 4.3）
```

## 十、关键设计决策回顾

### 决策1: 为什么 Jenkins 用 Inline Script 而非 SCM Pipeline？

```bash
# SCM Pipeline 期望通过 Git SCM 插件 clone → 读取 Jenkinsfile
# 但内网 Git Daemon (git:// 协议) 与 Jenkins SCM 插件兼容性差
# → 结果: "No flow definition, cannot run"

# Inline Script 直接把 Pipeline 代码写进 config.xml
# → 通过 sh 命令手动 git clone，绕开 SCM 插件
# → 代价: 脚本存储在 Jenkins 中，不在 Git 中（可通过 API 导出备份）
```

### 决策2: 为什么用 sed 而不是 kubectl？

```bash
# ❌ kubectl set image deploy/snownlp-demo app=xxx:v2
#   → ArgoCD 不知道这个变更，下次 sync 会回滚到 Git 中的旧版本

# ✅ sed -i "s|image: ...:.*|image: ...:v2|g" deployment.yaml
#   → git add && git commit && git push
#   → ArgoCD 检测到 Git diff → 自动同步 → Git = 唯一真相源
```

### 决策3: 为什么 monitoring-stack 的 prune=false？

```bash
# snownlp-demo: 业务应用，prune=true → Git 中删除文件，K8s 资源也删除
# monitoring-stack: 监控基础设施，prune=false → 误删 YAML 不会导致监控停服

# 具体配置:
kubectl patch application monitoring-stack -n argocd --type merge \
  -p '{"spec":{"syncPolicy":{"automated":{"prune":false}}}}'
```

### 决策4: 为什么用 ConfigMap 挂载 app.py 而不是构建到镜像？

```bash
# 镜像构建方式: 修改代码 → docker build(5min) → docker push(2min) → ArgoCD sync
# ConfigMap 方式: kubectl create configmap → kubectl rollout restart deploy(秒级)

# 代价: ConfigMap 有 1MB 大小限制，单文件场景适用

# 创建/更新 ConfigMap:
kubectl create configmap snownlp-demo-app \
  --from-file=app.py=01-app/app.py \
  -n monitoring --dry-run=client -o yaml | kubectl apply -f -

# 触发热更新:
kubectl rollout restart deployment/snownlp-demo -n monitoring
```
