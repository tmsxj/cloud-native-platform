## 一、Jenkins 相关问题

### 1.1 Pipeline（流水线） 状态 "No flow definition, cannot run"

**现象**：触发构建后立即失败，提示 "No flow definition, cannot run"

**根因**：Git 仓库是 bare repo（没有 working tree），SCM-based Pipeline 的 `checkout scm` 无法找到 `Jenkinsfile`。

**修复**：
```bash
# 将所有源文件推送到 Git 仓库
git push --force origin master:main

# 同时修复 Git Server 的 HEAD 引用
git symbolic-ref HEAD refs/heads/main
```

**预防**：确保 Git 仓库 init 时先 `git add .` 再 `git commit`，不要创建 bare repo。

---

### 1.2 Jenkins Reload 后 Job 404

**现象**：
- 通过 API 或直接写 `config.xml` 修改了 Job 配置
- 调用 `/reload` 后，Job 在 Jenkins UI 中消失（404 Not Found）
- 但 `config.xml` 文件仍在磁盘上

**根因**：Jenkins reload 时重新解析所有 `config.xml`。如果 XML 格式有问题（特殊字符未正确转义、编码问题等），Jenkins 会跳过该 Job 导致 404。

**修复**：
```bash
# 步骤1: 删除损坏的 Job 目录
rm -rf /var/jenkins_home/jobs/snownlp-demo-cicd

# 步骤2: 通过 API 重新创建（不要直接写文件！）
curl -X POST 'http://localhost:8080/createItem?name=snownlp-demo-cicd' \
  -H 'Jenkins-Crumb: CRUMB_TOKEN' \
  -H 'Content-Type: application/xml' \
  --data-binary @config.xml
```

**预防**：
- **永远通过 Jenkins API 创建/更新 Job**，不要直接操作 `config.xml`
- 如果需要更新脚本，用 API PUT 到 `/job/name/config.xml`
- XML 中的特殊字符（`<`、`>`、`&`）必须正确转义

---

### 1.3 IMAGE_TAG 显示 "null-BUILD_NUMBER"

**现象**：镜像标签变成 `null-3` 而不是 `20260625-101050-3`

**根因**：Inline Script Pipeline 中 `currentBuild.time` 不可用。`currentBuild.time.format()` 只在 SCM-based Jenkinsfile 中有效。

**修复**：
```groovy
// 错误写法（Inline Script 中不可用）
IMAGE_TAG = "${currentBuild.time.format('yyyyMMdd-HHmmss')}-${BUILD_NUMBER}"

// 正确写法（Groovy Date）
IMAGE_TAG = "${new Date().format('yyyyMMdd-HHmmss')}-${BUILD_NUMBER}"
```

### 1.4 Docker Build 超时 "Client.Timeout exceeded"

**现象**：
```
ERROR: failed to solve: python:3.9-slim: 
failed to resolve source metadata for docker.io/library/python:3.9-slim: 
Client.Timeout exceeded while awaiting headers
```

**根因**：内网/离线环境无法访问 Docker Hub。`FROM python:3.9-slim` 需要从 `docker.io` 拉取。

**修复**：
```dockerfile
# 改前（需要 Docker Hub）
FROM python:3.9-slim

# 改后（本地 Harbor）
FROM 192.168.1.61/library/python:3.11-slim
```

或在 Jenkinsfile 中自动修复：
```groovy
sed -i 's|FROM python:3.9-slim|FROM 192.168.1.61/library/python:3.11-slim|' Dockerfile
```

**前提**：确保 Harbor（私有镜像仓库） 中已有 `library/python:3.11-slim`：
```bash
curl -s http://192.168.1.61/api/v2.0/projects/library/repositories/python/artifacts
```

### 1.5 Git Clone 失败 "Connection refused"

**现象**：
```
ERROR: Error fetching remote repo 'origin'
fatal: unable to connect to git-server.git.svc.cluster.local:
git-server.git.svc.cluster.local[0: 10.109.128.228]: errno=Connection refused
```

**根因**：
- Git Daemon 端口 9418 不可达（跨 namespace 网络策略或 Pod（容器组） 正在重启）
- Git Server Pod 故障

**排查步骤**：
```bash
# 1. 检查 Git Server Pod 状态
kubectl get pods -n git

# 2. 检查 Git Service
kubectl get svc -n git

# 3. 从 Jenkins Pod 测试连通性
kubectl exec -n jenkins jenkins-xxx -- sh -c "nc -zv git-server.git.svc.cluster.local 9418"

# 4. 测试 git clone
kubectl exec -n jenkins jenkins-xxx -- sh -c "git ls-remote git://git-server.git.svc.cluster.local/repo.git"
```

**修复**：
- 等待 Git Server Pod 就绪
- 如果 Pod 频繁重启，检查资源限制和持久化存储
- 确保没有 NetworkPolicy 阻止跨 namespace 通信

## 二、ArgoCD（GitOps 持续交付工具） 相关问题

### 2.1 同步状态一直 "Unknown"

**现象**：ArgoCD UI 中 Application 状态显示 "Unknown"

**根因**：
- `repoURL` 为占位值（如 `https://your-git-server/...`）
- Git 仓库地址错误或不可达
- 仓库中路径 `path` 配置错误（目录不存在或为空）

**检查**：
```bash
# 查看 ArgoCD Application 详情
kubectl get application -n argocd snownlp-demo -o yaml

# 查看同步日志
kubectl logs -n argocd deployment/argocd-application-controller | grep snownlp-demo
```

**修复**：
```bash
# 更新 repoURL
argocd app set snownlp-demo --repo git://git-server.git.svc/snownlp-observability-demo.git

# 或直接编辑
kubectl edit application snownlp-demo -n argocd
```

---

### 2.2 ArgoCD 与 K8s（Kubernetes，容器编排引擎） 手动变更冲突

**现象**：手动 `kubectl edit deployment` 后，过几分钟被自动回滚

**根因**：配置了 `selfHeal: true`，这是 ArgoCD 的设计行为。

**说明**：`selfHeal: true` 意味着任何手动对 K8s 资源的修改都会被 ArgoCD 检测到并回滚到 Git 中定义的状态。这是 GitOps（以 Git 为唯一真相源的运维模式） 的核心原则——Git 是唯一真相源。

**正确做法**：通过修改 Git 中的 YAML 文件来变更，而不是直接操作 K8s：
```bash
# ❌ 错误：直接操作 K8s
kubectl edit deployment snownlp-demo -n monitoring

# ✅ 正确：修改 Git → ArgoCD 自动同步
vim 01-app/deployment.yaml
git add . && git commit -m "..." && git push
```

---

### 2.3 Prune 误删资源

**现象**：从 Git 仓库中删除了 YAML 文件，对应的 K8s 资源也被 ArgoCD 自动删除

**根因**：配置了 `prune: true`，ArgoCD 会自动清理 Git 中不存在的 K8s 资源。

**防护措施**：

1. 对重要基础设施（如 monitoring-stack）设置 `prune: false`
2. 使用 `PruneLast: true` 确保先创建新资源再删除旧资源
3. 使用 `finalizers: [resources-finalizer.argocd.argoproj.io]` 防止意外级联删除

**恢复误操作**：
```bash
# 从 Git 历史恢复被删文件
git checkout HEAD~1 -- 01-app/deployment.yaml
git commit -m "fix: restore deleted manifest" && git push
# ArgoCD 检测到后自动重建
```

## 三、Docker 镜像相关问题

### 3.1 Harbor Push 被拒绝

**现象**：
```
denied: requested access to the resource is denied
```

**排查**：
```bash
# 1. 确认 Harbor 登录状态
docker login 192.168.1.61 -u admin -p Harbor12345

# 2. 确认 Harbor 项目存在
curl -u admin:Harbor12345 http://192.168.1.61/api/v2.0/projects/monitoring

# 3. 如果项目不存在，先创建
curl -u admin:Harbor12345 -X POST http://192.168.1.61/api/v2.0/projects \
  -H "Content-Type: application/json" \
  -d '{"project_name":"monitoring","public":true}'
```

---

### 3.2 K8s 拉取镜像失败 "ImagePullBackOff"

**现象**：Pod 状态 `ImagePullBackOff`

**排查步骤**：
```bash
# 1. 确认 imagePullSecrets 存在
kubectl get secret harbor-regcred -n monitoring

# 2. 确认镜像 tags 存在
docker pull 192.168.1.61/monitoring/snownlp-demo:20260625-101050-2

# 3. 查看 Pod 事件
kubectl describe pod -n monitoring -l app=snownlp-demo
```

**Harbor regcred 创建方式**：
```bash
kubectl create secret docker-registry harbor-regcred \
  --namespace monitoring \
  --docker-server=192.168.1.61 \
  --docker-username=admin \
  --docker-password=Harbor12345
```

## 四、SkyWalking（APM 调用链追踪） 代理问题

### 4.1 启动方式错误（无调用链数据）

**现象**：Pod 运行正常但 SkyWalking UI 中看不到服务

**根因**：用 `python app.py` 或 `uvicorn app:app` 直接启动，没有通过 `sw-python` 包装器。Python SkyWalking Agent 需要 `sw-python run` 才能注入探针。

**修复**：
```yaml
# Deployment 中
command: ["sw-python", "run"]
args: ["/usr/local/bin/uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
```

**验证**：
```bash
# 进入 Pod，查看 Agent 是否加载
kubectl exec -n monitoring deployment/snownlp-demo -- env | grep SW_AGENT

# 查看 SkyWalking UI 中是否有 "snownlp-demo" 服务
# http://skywalking.lab.local:31716
```

---

### 4.2 OAP 连接失败

**现象**：SkyWalking UI 无数据，Pod 日志中有 gRPC 连接错误

**排查**：
```bash
# 1. 确认 SkyWalking OAP Service 存在
kubectl get svc skywalking-oap -n monitoring

# 2. 从业务 Pod 测试连通性
kubectl exec -n monitoring deployment/snownlp-demo -- sh -c "nc -zv skywalking-oap.monitoring 11800"

# 3. 检查 OAP Pod 状态
kubectl get pods -n monitoring -l app=skywalking-oap
```

## 五、Git 相关问题

### 5.1 Bare Repo HEAD 指向错误分支

**现象**：Git clone 成功但仓库为空（没有 working tree 文件）

**排查**：
```bash
# 检查 Git 仓库状态
git --git-dir=/git/repos/snownlp-observability-demo.git branch

# 检查 HEAD 指向
cat /git/repos/snownlp-observability-demo.git/HEAD
# 输出: ref: refs/heads/master  ← 错误！应该是 refs/heads/main
```

**修复**：
```bash
# 修正 HEAD 引用
git --git-dir=/git/repos/snownlp-observability-demo.git symbolic-ref HEAD refs/heads/main
```

## 六、快速诊断命令集

```bash
# ===== Jenkins 诊断 =====
# 检查 Jenkins Pod
kubectl get pods -n jenkins

# 查看最近构建状态
curl -s http://jenkins:8080/job/snownlp-demo-cicd/lastBuild/api/json | grep -E 'number|building|result'

# 查看构建日志
curl -s http://jenkins:8080/job/snownlp-demo-cicd/lastBuild/consoleText

# ===== ArgoCD 诊断 =====
# 检查 Application 状态
kubectl get application -n argocd snownlp-demo

# 手动触发同步
argocd app sync snownlp-demo --insecure --grpc-web

# 查看同步历史
argocd app history snownlp-demo --insecure --grpc-web

# ===== Harbor 诊断 =====
# 列出项目
curl -u admin:Harbor12345 http://192.168.1.61/api/v2.0/projects

# 查看特定镜像标签
curl -u admin:Harbor12345 \
  http://192.168.1.61/api/v2.0/projects/monitoring/repositories/snownlp-demo/artifacts

# ===== K8s 诊断 =====
# 查看 Deployment 镜像
kubectl -n monitoring get deploy snownlp-demo \
  -o jsonpath='{.spec.template.spec.containers[0].image}'

# 查看 Pod 日志
kubectl logs -n monitoring -l app=snownlp-demo --tail=50

# 查看所有 Pod 状态
kubectl get pods -n monitoring -o wide

# ===== Git 诊断 =====
# 从 Jenkins Pod 测试 Git 连通性
kubectl exec -n jenkins jenkins-xxx -- sh -c \
  "git ls-remote git://git-server.git.svc.cluster.local/snownlp-observability-demo.git"
```

## 七、常见问题速查表

| 症状 | 可能原因 | 解决方向 |
|------|---------|---------|
| Jenkins Job 404 | config.xml 损坏 / Reload 跳过 | 删除重创，通过 API 而非文件操作 |
| IMAGE_TAG=null | 用了 `currentBuild.time` | 改用 `new Date().format()` |
| Docker pull timeout | 内网无法访问 Docker Hub | 用 Harbor 基础镜像 |
| Git clone Connection refused | Git Server Pod 未就绪 | 等 Pod Ready / 检查 Service（服务，集群内服务发现） |
| ArgoCD Unknown | repoURL 占位 / Git 不可达 | 更新为实际 Git 地址 |
| ImagePullBackOff | harbor-regcred 不存在 / 错误 | 重建 K8s Secret（密钥对象） |
| SkyWalking 无数据 | 没用 sw-python run 启动 | 修改 Pod command |
| 手动 k8s 修改被回滚 | ArgoCD selfHeal=true | 通过 Git 修改而非 kubectl |
| Git push 被拒绝 | Git Server 只读 / 无权限 | 检查 Git Daemon 配置 |
| Pipeline 构建号错乱 | Job 被删除重建 | 构建号从 1 开始计数 |
