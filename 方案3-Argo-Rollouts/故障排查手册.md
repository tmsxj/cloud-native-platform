## 一、kustomize commonLabels 与 Rollout CRD 标签冲突

### 现象

```bash
kubectl apply -k k8s/overlays/staging

# 错误：
Error: Rollout "tomcat-app" is invalid:
spec.strategy.blueGreen.activeService: Invalid value "tomcat-app":
Service tomcat-app has unmatch label env in rollout
```

### 根因

kustomize 的 `commonLabels` 机制只对标准 Kubernetes 资源（Deployment、Service 等）生效。Rollout 是 CRD（Custom Resource Definition），kustomize 无法识别其内部结构。

具体来说：
- `commonLabels: env: staging` → Service selector 添加了 `env: staging` ✅
- `commonLabels: env: staging` → Rollout 的 `spec.selector.matchLabels` **没有** `env: staging` ❌
- `commonLabels: env: staging` → Rollout 的 `spec.template.metadata.labels` **没有** `env: staging` ❌

Argo Rollouts 在蓝绿部署时会验证：`activeService.spec.selector` 中的所有 label 必须出现在 Rollout 的 pod template labels 中。标签不匹配 → `InvalidSpec`。

### 排查

```bash
# 1. 查看 Service selector
kubectl get svc tomcat-app -n tomcat-staging -o jsonpath='{.spec.selector}' | jq
# 输出: {"app":"tomcat-app","env":"staging"}  ← 多了 env

# 2. 查看 Rollout pod template labels
kubectl get rollout tomcat-app -n tomcat-staging -o jsonpath='{.spec.template.metadata.labels}' | jq
# 输出: {"app":"tomcat-app"}  ← 没有 env！

# 3. diff 确认
diff <(kubectl get svc tomcat-app -n tomcat-staging -o json | jq -S .spec.selector) \
     <(kubectl get rollout tomcat-app -n tomcat-staging -o json | jq -S .spec.template.metadata.labels)
```

### 修复

**不推荐** `commonLabels`（对 Rollout 无效），改用显式 JSON patch：

```yaml
# overlays/<env>/kustomization.yaml
patches:
  - target:
      kind: Rollout
      name: tomcat-app
    patch: |-
      - op: add
        path: /metadata/labels/env
        value: "staging"
      - op: add
        path: /spec/template/metadata/labels/env
        value: "staging"
      - op: add
        path: /spec/selector/matchLabels/env
        value: "staging"
  - target:
      kind: Service
      name: tomcat-app
    patch: |-
      - op: add
        path: /metadata/labels/env
        value: "staging"
      - op: add
        path: /spec/selector/env
        value: "staging"
  - target:
      kind: Service
      name: tomcat-app-preview
    patch: |-
      - op: add
        path: /metadata/labels/env
        value: "staging"
      - op: add
        path: /spec/selector/env
        value: "staging"
```

**关键要点**：
- Rollout 需要 3 处 label：`metadata.labels`、`spec.template.metadata.labels`、`spec.selector.matchLabels`
- Service 需要 2 处：`metadata.labels`、`spec.selector`
- 蓝绿的 previewService 也需要同步添加

## 二、Secret 被 kustomize 占位符覆盖

### 现象

```bash
# 应用后 Pod 无法拉取镜像
kubectl get pods -n tomcat-dev
# NAME           READY   STATUS
# tomcat-app-xx  0/1     ImagePullBackOff

kubectl describe pod -n tomcat-dev tomcat-app-xx | grep -A2 "Failed"
# Failed to pull image ... x509: certificate signed by unknown authority
```

### 根因

`overlays/<env>/secret.yaml` 中的 Harbor 凭据字段是占位符：

```yaml
# secret.yaml 模板
data:
  .dockerconfigjson: REPLACE_WITH_BASE64
```

当 kustomization.yaml 的 `resources` 包含 `secret.yaml` 时，`kubectl apply -k` 会用这个占位符覆盖集群中已有的有效 Secret。

### 排查

```bash
kubectl get secret harbor-regcred -n tomcat-dev -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d
# 输出: REPLACE_WITH_BASE64   ← 被覆盖了！
```

### 修复

从 `resources` 列表注释掉 `secret.yaml`：

```yaml
# overlays/<env>/kustomization.yaml
resources:
  - ../../base
  - namespace.yaml
  # secret.yaml 已存在于集群，此处仅作参考模板
```

**最佳实践**：Secret 由集群管理员手动创建或通过 External Secrets Operator 管理，不应纳入 kustomize resources。

## 三、镜像 tag 已被 Harbor 垃圾回收清理

### 现象

```bash
kubectl get pods -n tomcat-dev
# NAME           READY   STATUS
# tomcat-app-xx  0/1     ImagePullBackOff

kubectl describe pod -n tomcat-dev tomcat-app-xx | grep -E "Failed|pull"
# Failed to pull image "192.168.1.61/tomcat-demo/tomcat-app:v1-init":
# manifest for 192.168.1.61/tomcat-demo/tomcat-app:v1-init not found
```

### 根因

Harbor 配置了垃圾回收策略，某次 GC 清理了 `v1-init` tag。

### 排查

```bash
# 检查 Harbor 中还有哪些 tag
curl -s -u admin:Harbor12345 \
  "https://192.168.1.61/api/v2.0/projects/tomcat-demo/repositories/tomcat-app/artifacts" \
  | jq '.[].tags[].name'
# 输出: ["latest"]   ← v1-init 已不存在

# 确认当前 Deployment/Rollout 使用的镜像
kubectl get rollout tomcat-app -n tomcat-dev -o jsonpath='{.spec.template.spec.containers[0].image}'
# 输出: 192.168.1.61/tomcat-demo/tomcat-app:v1-init   ← 问题所在
```

### 修复

```bash
# 方案1: 修改镜像 tag（推荐用于生产）
kubectl set image rollout/tomcat-app tomcat-app=192.168.1.61/tomcat-demo/tomcat-app:latest \
  -n tomcat-dev

# 方案2: 修改 base 配置后重新 apply
# 编辑 k8s/base/rollout.yaml → image: ...:latest
kubectl apply -k k8s/overlays/dev
```

**最佳实践**：使用 `imagePullPolicy: Always` + 语义化版本号（如 `v1.0.0`），避免依赖单一 tag。

## 四、旧 ReplicaSet 残留失败 Pod

### 现象

新 Rollout 已经 Healthy，但旧 ReplicaSet 的 Pod 反复 CrashLoopBackOff：

```bash
kubectl get pods -n tomcat-dev
# NAME                        READY   STATUS
# tomcat-app-xxx-yyy          1/1     Running          ← 新版 OK
# tomcat-app-599cc9b6d-aaa    0/1     CrashLoopBackOff  ← 旧版反复失败
# tomcat-app-599cc9b6d-bbb    0/1     ImagePullBackOff   ← 旧版反复失败
```

### 根因

金丝雀和蓝绿策略会保留旧 ReplicaSet 用于回滚。当旧 RS 使用的镜像 tag 不可用或配置有问题时，K8s 会持续尝试启动旧版 Pod → 反复失败。

### 排查

```bash
# 列出所有 RS
kubectl get rs -n tomcat-dev

# 检查旧 RS 的镜像
kubectl get rs -n tomcat-dev tomcat-app-599cc9b6d \
  -o jsonpath='{.spec.template.spec.containers[0].image}'
# 输出: 192.168.1.61/tomcat-demo/tomcat-app:v1-init   ← 问题镜像

# 查看当前 Rollout 使用的镜像（对比）
kubectl get rollout tomcat-app -n tomcat-dev \
  -o jsonpath='{.spec.template.spec.containers[0].image}'
# 输出: 192.168.1.61/tomcat-demo/tomcat-app:latest
```

### 修复

```bash
# 手动缩容旧 RS 到 0
kubectl scale rs tomcat-app-599cc9b6d -n tomcat-dev --replicas=0

# 如果是多环境，逐个处理
kubectl scale rs tomcat-app-6758f45cf9 -n tomcat-prod --replicas=0
```

> **注意**：不要删除旧 RS，它是 Rollout 回滚的基础。缩容到 0 即可停止 Pod 重建，同时保留 RS 元数据。

## 五、ArgoCD 自动同步回滚 Rollout

### 现象

`kubectl apply -k` 部署 Rollout 后，ArgoCD 检测到 OutOfSync → 自动 sync → 又变回 Deployment。

### 根因

ArgoCD Application 的 `repoURL` + `targetRevision` 指向的 Git 仓库中仍然是老版的 Deployment YAML，且 `syncPolicy.automated.selfHeal: true`。

### 排查

```bash
kubectl get app tomcat-app-staging -n argocd -o yaml | grep -A5 syncPolicy
# automated:
#   prune: true
#   selfHeal: true         ← 自动修复开着的
```

### 修复

```bash
# 方法1: 禁用 auto-sync（临时）
kubectl patch app tomcat-app-dev -n argocd --type=merge \
  -p '{"spec":{"syncPolicy":{"automated":null}}}'

# 方法2: 更新 ArgoCD Application source.path（永久）
kubectl patch app tomcat-app-dev -n argocd --type=merge \
  -p '{"spec":{"source":{"path":"argo-rollouts/k8s/overlays/dev"}}}'

# 将新配置 push 到 Git 仓库后恢复 auto-sync
kubectl patch app tomcat-app-dev -n argocd --type=merge \
  -p '{"spec":{"syncPolicy":{"automated":{"prune":true,"selfHeal":true}}}}'
```

## 六、蓝绿 preview Pod 不创建

### 现象

Rollout Healthy，但没有 preview Pod：

```bash
kubectl get pods -n tomcat-staging
# tomcat-app-xxx-yyy   1/1   Running    ← 只有 active
# tomcat-app-xxx-zzz   1/1   Running    ← 只有 active
# (没有 preview Pod!)
```

### 根因

蓝绿配置中缺少 `previewService` 或 `previewReplicaCount`：

```yaml
# 错误配置
strategy:
  blueGreen:
    activeService: tomcat-app
    # 缺少 previewService！
    autoPromotionEnabled: false
```

### 排查

```bash
# 查看 Rollout 策略
kubectl get rollout tomcat-app -n tomcat-staging -o yaml | grep -A10 strategy

# 检查是否有 previewService
kubectl get svc -n tomcat-staging | grep preview
# (无输出 → previewService 缺失)
```

### 修复

1. **创建 preview Service**：

```yaml
# overlays/<env>/preview-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: tomcat-app-preview
  labels:
    app: tomcat-app
spec:
  type: ClusterIP
  ports:
    - port: 8080
      targetPort: 8080
      name: http
  selector:
    app: tomcat-app
```

2. **Rollout 中配置 previewService**：

```yaml
# overlays/<env>/kustomization.yaml patches
strategy:
  blueGreen:
    activeService: tomcat-app
    previewService: tomcat-app-preview       # ← 必须配置
    previewReplicaCount: 1                    # ← 必须配置
    autoPromotionEnabled: false
```

3. **kustomization.yaml 中引用 preview Service**：

```yaml
resources:
  - ../../base
  - namespace.yaml
  - preview-service.yaml    # ← 添加这行
```

## 常用诊断命令

```bash
# Rollout 状态查看
kubectl argo rollouts list rollout -A                            # 全局列表
kubectl argo rollouts get rollout tomcat-app -n tomcat-dev --watch  # 实时观察
kubectl argo rollouts status tomcat-app -n tomcat-dev            # 状态摘要

# Rollout 详情
kubectl argo rollouts history tomcat-app -n tomcat-dev           # 发布历史
kubectl argo rollouts promote tomcat-app -n tomcat-staging       # 手动推进
kubectl argo rollouts abort tomcat-app -n tomcat-staging         # 中止发布

# 问题排查
kubectl describe rollout tomcat-app -n tomcat-dev                # 查看事件
kubectl get rs -n tomcat-dev --sort-by=.metadata.creationTimestamp  # RS 时间线
kubectl argo rollouts dashboard -n tomcat-dev                    # Web UI

# 镜像相关
kubectl get rollout -A -o json | jq '.items[] | {
  ns: .metadata.namespace,
  name: .metadata.name,
  image: .spec.template.spec.containers[0].image
}'
```
