# 01 · 升级计划（v1.28.15 → v1.29.15）

> 执行后已根据实际踩坑修正，下方每节末尾附「⚠️ 修正」。

## 一、镜像清单与仓库策略
kubeadm 升级控制面时，镜像默认从 `registry.k8s.io` 拉取。离线环境改为从 Harbor 拉，但**仓库路径必须和集群 kubeadm-config 的 `imageRepository` 一致**：

```
# 集群真实 imageRepository（kubeadm-config ConfigMap 第 22 行）：
192.168.1.61/registry.k8s.io
# → 控制面镜像须推到 Harbor 的 registry.k8s.io 项目，保持原始子路径：
192.168.1.61/registry.k8s.io/kube-apiserver:v1.29.15
192.168.1.61/registry.k8s.io/kube-controller-manager:v1.29.15
192.168.1.61/registry.k8s.io/kube-scheduler:v1.29.15
192.168.1.61/registry.k8s.io/kube-proxy:v1.29.15
192.168.1.61/registry.k8s.io/etcd:3.5.16-0
192.168.1.61/registry.k8s.io/coredns/coredns:v1.11.1   # coredns 自带子路径 coredns/coredns
192.168.1.61/registry.k8s.io/pause:3.9
```

- 精确清单由 `kubeadm config images list --kubernetes-version v1.29.15 --image-repository 192.168.1.61/registry.k8s.io` 得到（注意：这个 `--image-repository` 只在 `images list` 子命令有效，用于"预览"路径；真正升级时不要带它）。
- ⚠️ 修正：早期计划误写成推到 `kubernetes` 项目 + `kubeadm upgrade apply --image-repository=...`。实际 `kubeadm upgrade apply` **不接受** `--image-repository`（unknown flag），且项目名也错。正确做法：镜像推 `registry.k8s.io` 项目，apply 不带该 flag，靠 ConfigMap 的 imageRepository 自动取用。

## 二、二进制备料（kubeadm / kubelet / kubectl）
离线节点无法 `apt-get install`，改用官方预编译二进制：
- 源：`https://dl.k8s.io/v1.29.15/kubernetes-server-linux-amd64.tar.gz`（369M，含全部 server 二进制；md5 `cb5e65c5d6521818be90b8a1714872f9`）
- 链路：US 下载 → H1(`harbor`) → **Windows 宿主机作中继** scp 到 m1~m3/w1/w2 的 `/tmp`（节点间无互信 SSH）
- 安装：解压后把 kubeadm/kubelet/kubectl 放到 `/usr/bin`；**cp 之后务必 `chmod 755 /usr/bin/kubelet /usr/bin/kubeadm /usr/bin/kubectl`**（源二进制在 /tmp 是无执行位的 `-rw-r--r--`，直接 cp 会让 /usr/bin/kubelet 丢 x 位，导致 systemd 无法启动 kubelet，升级 etcd 阶段超时回滚）
- ⚠️ 修正：`kubeadm upgrade node --control-plane` 在 1.29 **已移除该 flag**，直接 `kubeadm upgrade node` 即可（自动识别控制面节点升级静态 Pod）。

## 三、升级前清理（用户要求：非必要负载全缩 0）
升级前把非核心工作负载副本置 0，降低风险、加快升级：
- 双网格：`linkerd-demo`、`istio-demo`（注入 sidecar 随之缩 0）、控制面 `istio-system` / `linkerd` / `linkerd-viz` 也临时缩 0
- 其余"全家桶"（argo/argocd/cert-manager/chaos/gitlab/keda/monitoring/tomcat/velero 等）升级前本就处于聚焦模式 `scale 0`，不动
- **关键坑预警（已验证必须做）**：仅 scale 0 不够 —— 这些负载的 **mutating/validating webhook**（Linkerd injector、Istio injector）配置仍在，升级时 apiserver 创建/更新系统 Pod 会被 webhook 拦截且因后端 Pod 已停而超时卡死。因此需**同时删除相关 webhook configuration**（记录清单，升级后按需重建）。
- ⚠️ 修正：删掉的注入 webhook **不会被 istiod / linkerd-injector 自动重建**，升级后必须用原安装清单重放（见六）。

## 四、升级顺序（kubeadm 标准）
1. **m1（首个控制面）**：装新 kubeadm → `kubeadm upgrade plan` → `kubeadm upgrade apply v1.29.15`（不带 `--image-repository`）→ 升级本机 kubelet/kubectl（cp + chmod 755）→ 重启 kubelet
2. **m2 / m3（其余控制面）**：cp+chmod 三二进制 → 重启 kubelet → `kubeadm upgrade node`（不带 `--control-plane`）
3. **w1 / w2（工作节点）**：cp+chmod 三二进制 → 重启 kubelet → `kubeadm upgrade node`（自动跳过控制面阶段）

> 版本偏差规则：控制面必须先于节点升级（kubelet 不能领先 apiserver 超过一个小版本）。
