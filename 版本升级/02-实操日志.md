## 阶段 1 · 离线物料备料（✅）

### 1.1 kubeadm/kubelet/kubectl 二进制
- 经 US 下载 `https://dl.k8s.io/v1.29.15/kubernetes-server-linux-amd64.tar.gz`（369M，md5 `cb5e65c5d6521818be90b8a1714872f9`）→ 落 H1 `/tmp` → scp 到 m1 `/tmp`。

- **节点间无互信 SSH**：m1 不能直连 m2/m3/w1/w2。改由 **Windows 宿主机作中继**（ssh 别名 `m1~m3/w1/w2` 可用，裸 IP 会挂起超时）→ 把 3 个二进制 scp 到每个节点的 `/tmp`。

### 1.2 控制面镜像入 Harbor（私有镜像仓库） `registry.k8s.io` 项目
- 脚本：`scripts/sync-k8s-controlplane-images.sh`（US `docker pull`+`save` → H1 `scp` → H1 `load`+`tag`+`push` 到 Harbor 的 **`registry.k8s.io`** 项目，保持原始子路径）。

### 坑 1（已解决）：`sync_from_us.ps1` 在当前 PowerShell Core 下直接报错
- **现象**：`powershell -File sync_from_us.ps1 ...` 报一堆 `Missing opening '(' after keyword 'for'`、`'||' is not a valid statement separator` 等 ParserError，脚本根本没执行下载。
- **根因**：该脚本把 bash 代码写在 PowerShell 可展开 here-string(`@" ... "@`)里给 US/H1 执行；当前默认 `powershell` 是 **PowerShell Core**，here-string 内 bash 语法（`for/if/||/do`）被当成 PowerShell 语法解析失败。脚本是给 PowerShell 5.1 写的。
- **解决**：**不改动用户脚本**（它在 gitignore 的本地目录里），绕过它，直接复用已验证通的 `H1 → US` 链路写等价 bash 流程。tarball 与控制面镜像都走这条路径，已验证可用。

### 坑 2（已解决）：错推到 Harbor `kubernetes` 项目
- **现象**：早期按"计划"把 7 个控制面镜像推到了 Harbor 的 **`kubernetes`** 项目（`192.168.1.61/kubernetes/...`），后来 `kubeadm upgrade apply` 报找不到 `192.168.1.61/registry.k8s.io/kube-apiserver`。
- **根因**：集群 kubeadm-config ConfigMap（配置字典） 的 `imageRepository` 真实值是 **`192.168.1.61/registry.k8s.io`**（离线部署时就是这样设的），不是 `kubernetes`、也不是默认 `registry.k8s.io`。
- **解决**：用 `scripts/sync-k8s-controlplane-images.sh` 把 7 个镜像**重新推到 `registry.k8s.io` 项目并保持原始子路径**（`registry.k8s.io/coredns/coredns`、`registry.k8s.io/etcd` 等）。ConfigMap 不动。

## 阶段 3 · 控制面升级 m1（✅）

1. `kubeadm upgrade plan` → 显示 `1.28.0 → 1.29.15` 升级路径可用。
2. **坑 3（已解决）：`kubeadm upgrade apply` 不支持 `--image-repository`**
   - **现象**：`kubeadm upgrade apply v1.29.15 --image-repository=192.168.1.61/kubernetes` → `unknown flag: --image-repository`。
   - **根因**：`apply` 子命令根本没有该 flag（它从 ConfigMap 读 imageRepository）。早期计划误记。
   - **解决**：去掉该 flag，镜像仓库完全由 ConfigMap 的 `imageRepository` 决定。
3. **中间弯路**：曾试图 `sed` 把 ConfigMap 的 imageRepository 改成 `192.168.1.61/kubernetes`（顺着坑 2 的误判），`apply` 仍报找不到 `192.168.1.61/registry.k8s.io/kube-apiserver` —— 反过来印证了真实值是 `registry.k8s.io`。ConfigMap 改回，**保持原值不动**。
4. **坑 2 收尾**：确认 `kubeadm-config` 第 22 行 `imageRepository: 192.168.1.61/registry.k8s.io`，coredns 路径 `192.168.1.61/registry.k8s.io/coredns`；containerd `sandbox_image` 也是 `.../registry.k8s.io/pause:3.9`。据此用 `scripts/sync-k8s-controlplane-images.sh` 把镜像重推到 `registry.k8s.io` 项目。
5. **成功**：`kubeadm upgrade apply v1.29.15 --yes`（不带 flag）→ `rc=0`，m1 etcd/apiserver/controller/scheduler 静态 Pod（容器组） 全部升级；重启 m1 kubelet。验证 `kube-apiserver-master1` 1/1 Running。

## 阶段 4 · 节点升级 m2 / m3 / w1 / w2（✅）

> 通用脚本 `scripts/node-upgrade.sh`：`cp` 三二进制 → **`chmod 755`** → 重启 kubelet → `kubeadm upgrade node`（**不带 `--control-plane`**）。二进制经 Windows 中继 scp 到各节点 `/tmp` 后 `sudo bash` 执行。

### 坑 4（已解决）：`kubeadm upgrade node --control-plane` 在 1.29 已移除
- **现象**：`kubeadm upgrade node --control-plane` → `unknown flag: --control-plane`。
- **根因**：kubeadm v1.29 的 `upgrade node` 不再接受该 flag，会自动检测本节点是否为控制面并升级静态 Pod（phases 含 `control-plane`/`kubelet-config`），工作节点则自动跳过控制面阶段。
- **解决**：一律用 `kubeadm upgrade node`，无需区分控制面/工作节点。

### 坑 5（致命 · 已解决）：`cp` 覆盖 kubelet 丢失执行位 → systemd 无法启动 kubelet → etcd 阶段超时回滚
这是本次最阴险的坑，单独记详：
- **现象**：m2 第一次升级 `kubeadm upgrade node` 在 `[upgrade/etcd]` 阶段卡 5 分钟 → `static Pod hash for component etcd on Node（节点） master2 did not change after 5m0s: timed out waiting for the condition` → 自动**回滚** etcd 到升级前状态，整步 `rc=1`。
- **排查**：`systemctl status kubelet` 显示 `activating (auto-restart)`；`journalctl -u kubelet` 明文：
  ```
  kubelet.service: Failed to locate executable /usr/bin/kubelet: Permission denied
  kubelet.service: Failed at step EXEC spawning /usr/bin/kubelet: Permission denied
  ```
  `ls -l /usr/bin/kubelet` 显示 `-rw-r--r--`（**没有 x 位**）。
- **根因**：`cp /tmp/kubelet /usr/bin/kubelet` 时，源文件 `/tmp/kubelet` 本身是 `-rw-rw-r--`（无 exec 位，kubeadm/kubectl 之前拷贝时保留了执行位所以不报错，唯独 kubelet 丢了）。`cp` 覆盖后目标失去 x 位 → systemd 无法 exec kubelet → kubelet 崩溃重启循环（status 203/EXEC）。此时 etcd/apiserver 等静态 Pod 是崩溃前的旧 kubelet 留下的孤儿进程仍在跑，但**新 kubelet 没在托管**，于是 `kubeadm upgrade node` 写入的新 etcd 清单无人接管、hash 始终不变 → 超时回滚。
- **解决**：`chmod 755 /usr/bin/kubelet`（顺手把 kubeadm/kubectl 也 chmod 一遍）→ kubelet `active`、版本 `v1.29.15` → 重跑 `kubeadm upgrade node` → etcd/apiserver/controller/scheduler 全部 `upgraded successfully`，`rc=0`。
- **固化到脚本**：`scripts/node-upgrade.sh` 的 `cp` 之后**强制 `chmod 755`**，从此 m3/w1/w2 一路绿灯。

### 结果
- m2：修复执行位后升级成功（etcd/apiserver/controller/scheduler 全 upgraded）。
- m3（控制面）：`kubeadm upgrade node` 走完 etcd→apiserver→controller→scheduler，并 `Applied essential addon: CoreDNS / kube-proxy`，`rc=0`。
- w1 / w2（工作节点）：`Skipping prepull. Not a control plane node.` / `Skipping phase. Not a control plane node.`，仅更新 kubelet 配置，`rc=0`。
- 三个节点并行执行，全部 `kubelet active`、版本 `v1.29.15`。
