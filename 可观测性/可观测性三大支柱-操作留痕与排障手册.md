## 3. 操作步骤详解

### 3.1 Metrics: Prometheus + AlertManager

#### 3.1.1 告警规则部署

**操作**：将 `prometheus-alerting-rules.yml` 注入 `prometheus-server` ConfigMap（配置字典） → 重启 Prometheus

```bash
# 1. 将告警规则 YAML 注入 ConfigMap
kubectl create configmap prometheus-server -n monitoring \
  --from-file=alerting_rules.yml=prometheus-alerting-rules.yml \
  --dry-run=client -o yaml | kubectl replace -f -

# 2. 重启 Prometheus 加载新规则
kubectl rollout restart deployment/prometheus-server -n monitoring
```

**15 条规则覆盖范围**：

| 分组 | 规则数 | 严重级别 | 说明 |
|------|--------|---------|------|
| node-alerts | 7 | warning/critical | CPU>80%/95%、内存>85%/95%、磁盘耗尽、磁盘>85%、文件系统只读 |
| pod-alerts | 3 | warning | CrashLooping、NotReady、Pending |
| deployment-alerts | 1 | warning | 副本数不匹配 |
| statefulset-alerts | 1 | warning | 副本异常 |
| pvc-alerts | 1 | warning | 可用空间<15% |
| prometheus-alerts | 2 | critical | 目标下线、停止摄入样本 |
| watchdog | 1 | none | 心跳检测（始终触发，验证管道通） |

#### 3.1.2 AlertManager 路由配置

**操作**：注入 `alertmanager.yml` → 重启 AlertManager

```bash
kubectl create configmap alertmanager -n monitoring \
  --from-file=alertmanager.yml=alertmanager-email.yml \
  --dry-run=client -o yaml | kubectl replace -f -

kubectl rollout restart statefulset/alertmanager -n monitoring
```

**路由结构**：

```
告警流入
  ├─ severity=critical → email-critical  (主题前缀 [CRITICAL])
  ├─ severity=warning  → email-warning   (主题前缀 [WARNING])
  └─ 其他              → email-default   (主题前缀 [ALERT])
```

**抑制规则**：
- `NodeCPUCritical` 触发时 → 抑制同节点所有 `KubePod*` 告警
- `NodeCPUCritical` 触发时 → 抑制同节点 `NodeDiskPressure`

#### 3.1.3 邮件验证

配置 QQ 邮箱 SMTP (`smtp.qq.com:587`)：

```yaml
global:
  smtp_smarthost: smtp.qq.com:587
  smtp_from: tmsxj@foxmail.com
  smtp_auth_username: tmsxj@foxmail.com
  smtp_auth_password: <QQ邮箱授权码>
  smtp_require_tls: true
```

**验证结果**：收到 4 封邮件，三级路由全部验证通过。

### 3.2 Logs: Promtail + Loki

#### 3.2.1 日志采集链路

```
应用 stdout → CRI 运行时 → /var/log/pods/<ns>_<pod>_<uid>/<container>/0.log
                                           ↓
                           Promtail DaemonSet (每节点一个) 读取
                                           ↓
                           CRI 格式解析 → 正则提取 namespace/pod_name
                                           ↓
                           正则提取 trace_id → 推送 Loki
```

#### 3.2.2 Promtail 配置关键点

```yaml
pipeline_stages:
  - cri: {}                              # 阶段1: 解析容器运行时格式
  - regex:                               # 阶段2: 从文件路径提取元数据
      expression: '^/var/log/pods/(?P<namespace>[^_]+)_(?P<pod_name>[^_]+)_...'
      source: filename                   # 从文件名提取，不受日志内容影响
  - labels:                              # 提升为 Loki label
      namespace:
      pod_name:
  - regex:                               # 阶段3: 提取 trace_id
      expression: 'trace_id=(?P<trace_id>\w+)'
  - labels:
      trace_id:
```

#### 3.2.3 系统日志扩展（2026-06-24 新增）

**为什么补这步**：之前的 Promtail 只采了 `/var/log/pods`（业务容器日志），基础设施层的日志（kubelet、docker daemon、系统内核消息）完全漏掉了。出了问题只能看业务日志，排查不到根因。

**操作步骤**：

```bash
# 1. DaemonSet 新增 hostPath: /var/log → 容器内 /var/log/host (readOnly)
#    见 promtail-daemonset.yaml，关键新增：
#    volumes:
#      - name: varlog
#        hostPath:
#          path: /var/log
#    volumeMounts:
#      - name: varlog
#        mountPath: /var/log/host
#        readOnly: true

# 2. Promtail ConfigMap 新增 3 个 scrape job
kubectl apply -f promtail-config.yaml    # 已更新配置

# 3. 滚动重启 DaemonSet 使 volume mount 和配置生效
kubectl rollout restart daemonset/promtail -n monitoring

# 4. 验证
kubectl logs -n monitoring daemonset/promtail | grep "system-logs\|host-logs\|kubelet-logs\|docker-daemon"
```

**新增 3 个采集 job**：

| Job | 路径 | 采集内容 | 价值 |
|-----|------|---------|------|
| `host-logs` | `/var/log/host/{messages,syslog}` | 内核消息、systemd、OOM killer | 排查节点级故障根因 |
| `kubelet-logs` | `/var/log/host/kubelet*.log` | kubelet 启动/重启/镜像拉取日志 | Pod 无法调度的第一现场 |
| `docker-daemon-logs` | `/var/log/host/docker*.log` | Docker 自身运行状态 | 镜像拉取失败、存储驱动报错 |

**注意事项**：
- 多数 kubeadm 集群 kubelet/containerd 走 **journald**，文件可能采不到
  - 补齐方案：Promtail 的 `journal` stage + 挂载 `/run/log/journal`
  - 当前不做实操，保留作为已知缺口
- `/var/log/host` 使用独立挂载点，不跟 `/var/log/pods` 冲突（路径前缀不同）

### 3.3 Traces: SkyWalking + Elasticsearch

#### 3.3.1 应用端埋点

**核心代码**（`app.py`）：

```python
# 1. Entry Span 手动创建
@app.middleware("http")
async def sw_trace_middleware(request, call_next):
    ctx = get_context()
    with ctx.new_entry_span(op=f"{request.method}:{request.url.path}") as span:
        span.layer = Layer.Http   # 标注 HTTP 层，UI 拓扑图才能正确展示
        response = await call_next(request)
        return response

# 2. trace_id 日志注入
class SafeSWFilter(logging.Filter):
    def filter(self, record):
        ctx = get_context()
        if ctx and ctx.segment:
            record.trace_id = str(ctx.segment.related_traces[0])
        else:
            record.trace_id = "N/A"     # 安全降级
        return True

logging.basicConfig(format='%(asctime)s [%(levelname)s] trace_id=%(trace_id)s %(message)s')
logger.addFilter(SafeSWFilter())
```

**启动方式**：`sw-python run uvicorn app:app --host 0.0.0.0 --port 8000`（不能用 `python app.py`）

#### 3.3.2 存储后端

SkyWalking OAP → Elasticsearch (8.11.0)，gRPC 端口 11800

### 3.5 三支柱持久化补齐

#### 3.5.1 问题发现

审计 monitoring 命名空间 PVC 列表时发现：
- Loki StatefulSet 使用 `emptyDir`
- Elasticsearch StatefulSet 使用 `emptyDir`
→ Pod 重启后日志和 trace 数据全部丢失

#### 3.5.2 修复步骤

```bash
# 1. 创建 PVC
kubectl apply -f 可观测性/fixes/loki-pvc.yaml
kubectl apply -f 可观测性/fixes/elasticsearch-pvc.yaml

# 2. Patch StatefulSet 替换 emptyDir 为 PVC
#    - Loki: volumeClaimTemplates 挂载 /var/loki
#    - ES: volumeClaimTemplates 挂载 /usr/share/elasticsearch/data

# 3. 重启 Pod
kubectl rollout restart statefulset/loki -n monitoring
kubectl rollout restart statefulset/elasticsearch -n monitoring
```

#### 3.5.3 最终 PVC 清单

| PVC | 大小 | 用途 | 物理路径 |
|-----|------|------|---------|
| prometheus-server | 50Gi | TSDB | worker1 `/opt/local-path-provisioner/...` |
| grafana | 10Gi | 仪表盘 | worker1 |
| loki-storage | 5Gi | 日志 | worker2 |
| elasticsearch-data | 10Gi | trace | worker1 |
| storage-alertmanager-0 | 50Mi | 静默 | — |

#### 3.5.4 逻辑存储与物理存储的关系

> **重要理解**：Loki 不是"本地临时文件"，它有自己的 BoltDB 索引 + 压缩 chunk 存储引擎。
> 与 ES 的区别在于设计哲学——Loki 只索引 label 不索引正文，省磁盘但全文搜索慢；
> ES 索引每一个词，搜索快但占空间大。两者各司其职：
> - **Loki** 存日志（按 namespace/pod + 时间范围 翻页查看）
> - **ES** 存 trace Span（需要聚合、排序、字段过滤的结构化查询）

## 4. 排障实录

### 4.1 Prometheus 告警规则不生效

**现象**：Prometheus UI 的 Alerts 页面一片空白，没有任何告警。

**排查过程**：
1. `kubectl exec prometheus-server-xxx -- ls /etc/config/` → 确认 `alerting_rules.yml` 文件存在
2. 检查 Prometheus 日志 → 无报错
3. 回到 ConfigMap 对比 → 发现 `prometheus-server` ConfigMap 中 `rule_files` 路径配置了多个但实际只有 `/etc/config/alerting_rules.yml`

**根因**：ConfigMap 用 `kubectl create configmap` 全量替换时，需要同时把所有文件塞进去——但实际操作中 `prometheus-alerting-rules.yml` 这个独立文件没有被挂载进去。

**修复方案**：用 `--from-file` 将 YAML 文件重新打包进 ConfigMap，然后 `kubectl rollout restart` Prometheus。

**教训**：改 Prometheus ConfigMap 时不要手动 patch，一次性 `kubectl create --dry-run ... | kubectl replace` 保证完整性。

---

### 4.2 AlertManager 配置反复重载

**现象**：更新 AlertManager ConfigMap 后，规则似乎没生效，路由行为与预期不符。

**排查过程**：
1. AlertManager UI 显示旧配置 → 确认 reload 未生效
2. 发现 AlertManager 自动 reload 不是即时的，需要数十秒
3. 多次 `kubectl rollout restart` 后发现配置是纯被旧 ConfigMap 覆盖了

**根因**：AlertManager StatefulSet 挂载 ConfigMap 时用了 subPath，子路径模式不支持自动热更新 ConfigMap 内容——必须 rolling restart。

**修复方案**：每次修改 alertmanager ConfigMap 后，必须执行 `kubectl rollout restart statefulset/alertmanager -n monitoring`。

**教训**：StatefulSet + subPath 挂载 → 永远重启。只有 Deployment（部署，无状态工作负载） + 全目录挂载才能热加载。

---

### 4.3 QQ 邮箱 SMTP 授权码配置

**现象**：AlertManager 日志报 `authentication failed`，邮件发不出去。

**排查过程**：
1. 确认 `smtp.qq.com:587` 可达（Prometheus → AlertManager → QQ SMTP 网络通）
2. 检查密码字段 → 使用的不是 QQ 密码，而是**授权码**
3. QQ 邮箱设置 → 账户 → POP3/SMTP 服务 → 生成的 16 位授权码

**根因**：QQ 邮箱从 2020 年起禁用密码登录第三方客户端，必须使用独立 SMTP 授权码。

**修复方案**：在 AlertManager ConfigMap 中 `smtp_auth_password` 填写授权码（非密码）。

**验证**：重启后收到 Watchdog 心跳告警邮件 ✅

**教训**：国内邮箱（QQ/163/126）一律不能拿登录密码配 SMTP，必须进邮箱后台生成授权码。

### 4.4 Helm 不可用——kubectl 原生平替

**现象**：`helm install` 或 `helm upgrade` 报错，Helm 客户端无法正常使用。

**排查过程**：
1. 物理机能访问外网（代理配置），但 VM 内 Helm 连不到仓库
2. 尝试 `helm repo update` → 超时

**根因**：集群环境网络限制，Helm 的 chart 仓库拉取失败（代理未在 VM 内配置或 repo 地址不可达）。

**修复方案**：全程使用 kubectl 原生方式替代 Helm：
- 不用 `helm install` → 手写 Deployment + ConfigMap + Service YAML
- 不用 `helm upgrade` → `kubectl apply -f` + `kubectl rollout restart`
- 创建 PVC、patch StatefulSet 均用 `kubectl`

**教训**：生产环境不一定有 Helm，熟练掌握 kubectl 原生操作是运维基本功。Helm 只是 YAML 模板引擎的便利层——能不能撇开 Helm 干活，是区分熟练度的分水岭。

### 4.5 Loki/ES 无持久化存储

**现象**：审计时发现 Loki 和 Elasticsearch 的 StatefulSet 中 `volumes` 声明的是 `emptyDir: {}`。

**排查过程**：
1. `kubectl get statefulset loki -o yaml` → `volumes: - name: storage - emptyDir: {}`
2. `kubectl get statefulset elasticsearch -o yaml` → 同样是 `emptyDir`

**根因**：初期部署时省略了 PVC 配置，所有数据写临时目录。Pod 重启后日志（Loki）和 trace（ES）全部丢失。**这个问题在"跑起来就行"阶段被忽略了。**

**修复方案**：
1. 创建 PVC `loki-storage` (5Gi) 和 `elasticsearch-data` (10Gi)
2. 用 `kubectl patch statefulset` 以 JSON patch 语法替换 emptyDir 为 PVC
3. `kubectl rollout restart` 重建 Pod

**Patch JSON 示例**：
```json
[
  {"op": "remove", "path": "/spec/template/spec/volumes/0/emptyDir"},
  {"op": "add", "path": "/spec/template/spec/volumes/0/persistentVolumeClaim", "value": {"claimName": "loki-storage"}}
]
```

**教训**：部署阶段就要检查存储类型——`emptyDir` 只能用于临时数据，持久化数据必须配 PVC。配置完毕后用 `kubectl get pvc -A` 全局审计一遍。

### 4.6 Loki ruler 日志告警未配置

**现象**：Loki 支持内置 ruler 模块触发日志告警，但 `alertmanager_url` 为空串 `""`。

**排查过程**：
1. `kubectl get configmap loki-config -o yaml` → 发现 `ruler.alertmanager_url: ""`
2. 这意味着即使配了 Loki 告警规则，也无法推送到 AlertManager

**根因**：Loki 初始部署时 ruler 模块写了空值占位，后续补齐 Prometheus 告警时没回去修 Loki 配置。

**修复方案**：patch Loki ConfigMap：
```yaml
ruler:
  alertmanager_url: http://alertmanager.monitoring:9093
```
然后 `kubectl rollout restart statefulset/loki -n monitoring`

**教训**：三支柱中 Logs 这支柱最容易漏告警——大家都记得 Prometheus 要配 rule，但 Loki 日志告警往往被忽略。趁早把 `alertmanager_url` 配上，后面写规则就水到渠成了。

### 4.7 trace_id 注入不生效

**现象**：应用日志中 trace_id 总是显示 `N/A`，Loki 中查不到 `trace_id` label。

**排查过程**：
1. `kubectl logs snownlp-demo-xxx` → 每行日志 `trace_id=N/A`
2. 检查代码逻辑 → `SafeSWFilter.filter()` 逻辑正确
3. 检查容器启动命令 → 用的是 `python app.py` 而非 `sw-python run`

**根因**：`python app.py` 直接启动时，`import skywalking` 虽然不报错但 Agent 没有初始化——`get_context()` 始终返回空对象。

**修复方案**：Dockerfile 中启动命令改为：
```dockerfile
ENTRYPOINT ["sw-python", "run"]
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
```
`sw-python` 是 SkyWalking Python Agent 的包装器，会自动注入探针到进程，拦截所有 HTTP/gRPC 调用。

**教训**：SkyWalking Python Agent 不是 pip install 就完事——必须用 `sw-python run` 启动才能注入探针。这个坑在官方文档里有写，但很容易被跳过。

### 4.8 CRI 日志 pipeline 顺序错误

**现象**：Loki 中查不到 `pod_name` 和 `namespace` label。

**排查过程**：
1. 检查 Promtail 自身日志 → 无报错
2. 单步调试 pipeline → 发现 CRI 解析后日志格式变了，路径正则匹配的位置不对
3. 调整 pipeline_stages 顺序后生效

**根因**：Promtail pipeline 阶段是按顺序执行的。如果先 `regex` 提取路径元数据，再 `cri` 解析——CRI 解析会把原始日志格式变掉，导致后续阶段拿不到原始文件路径信息。

正确顺序必须是：
```yaml
pipeline_stages:
  - cri: {}          # 先解析 CRI 格式
  - regex: ...       # 再从文件路径提取 namespace/pod
  - regex: ...       # 最后从日志内容提取 trace_id
```

**教训**：Log pipeline 的顺序就是处理链——数据进管道后，"预处理"操作必须在"提取"操作之前。这和 ETL 的 Extract-Transform-Load 顺序是反的（这里是 T→E→L），写配置时容易搞混。

### 4.9 derivedFields 点击链接不出现

**现象**：Grafana Loki 面板中日志行正常显示，但 `trace_id=xxx` 没有变成可点击链接。

**排查过程**：
1. Grafana Data Sources → Loki → 确认 derivedFields 配置已保存
2. Explore 中查看原始日志 → `trace_id` 确实存在但未渲染为链接
3. 排查 Loki 端 → Promtail 把 trace_id 提取为 label，但 Loki 没有落地结构化元数据

**根因**：Loki 配置中 `allow_structured_metadata: true` 未开启。默认情况下 Loki 不存储 Promtail 提取的自定义 label——即使 Promtail 正确提取了 trace_id，推送到 Loki 后也被丢弃。

**修复方案**：在 Loki 配置中开启：
```yaml
allow_structured_metadata: true
```
然后重启 Loki → 新日志中的 trace_id 正常落地 → Grafana derivedFields 自动渲染为链接。

**教训**：日志关联追踪的链路有三个环节：① 应用代码注入 trace_id ② Promtail 提取 ③ Loki 存储 → Grafana 渲染。每个环节都可能断。排查时要逐个环节验证——先看应用有没有 trace_id，再看 Promtail 提没提取，最后看 Loki 存没存。

### 4.10 uvicorn 日志重复输出

**现象**：每条请求日志在 Loki 中出现两次，一次是自定义格式（带 trace_id），一次是 uvicorn 默认格式（无 trace_id）。

**排查过程**：
1. 对比两条日志的 source → 来自同一个 Pod、同一个容器
2. 检查代码 → `uvicorn.access` logger 添加了自定义 handler，但 uvicorn 启动时自带一个默认 handler

**根因**：uvicorn 启动时自动配置了 `uvicorn.access` logger 的 StreamHandler（默认格式）。代码中添加自定义 handler 时没有清掉旧的，导致一条访问日志被两个 handler 各输出一次。

**修复方案**：
```python
uvicorn_logger = logging.getLogger("uvicorn.access")
uvicorn_logger.handlers.clear()   # 必须先清除默认 handler！
handler = logging.StreamHandler()
handler.setFormatter(...)
uvicorn_logger.addHandler(handler)
uvicorn_logger.addFilter(SafeSWFilter())
```

**教训**：Python logging 框架中，logger 是树形结构的，根 logger 和子 logger 都可以加 handler。uvicorn 框架本身会注册 handler，覆盖配置前必须 `.handlers.clear()`，否则就是"追加"而非"替换"。

### 4.11 容器启动失败 sw-python 报错

**现象**：`sw-python run uvicorn ...` 启动后立即 CrashLoopBackOff。

**排查过程**：
1. `kubectl logs snownlp-demo-xxx` → `ModuleNotFoundError: No module named 'skywalking'`
2. `kubectl exec snownlp-demo-xxx -- pip list | grep skywalking` → 确认已安装
3. 检查 Dockerfile 安装顺序 → `apache-skywalking` 后安装，依赖关系可能有问题

**根因**：`apache-skywalking` 的 pip 安装需要依赖 `fastapi`，但一次性 `pip install -r requirements.txt` 中 alpah-skywalking 如果排在前面，依赖解析会失败。

**修复方案**：Dockerfile 分两步安装：
```dockerfile
RUN pip install --no-cache-dir -r requirements.txt \
    && pip install --no-cache-dir apache-skywalking
```
先装好 fastapi/uvicorn/prometheus_client，再装 skywalking agent。

**教训**：pip 的依赖解析在 Docker 构建时是线性的——如果包 A 依赖包 B，但 pip 先处理 A 时 B 还没装，会有隐性问题。分步安装是稳妥做法。

## 5. 最终架构与数据流

### 5.1 四条数据链路

```
链路① 请求 → Trace
  用户请求 → Ingress → snownlp-demo → sw_trace_middleware → gRPC → SkyWalking OAP → ES

链路② 请求 → 日志
  应用 stdout → CRI → /var/log/pods → Promtail → Loki (PVC 5Gi)

链路③ 请求 → 指标
  /metrics 端点 → Prometheus scrape → TSDB (PVC 50Gi) → Grafana Dashboard

链路④ 日志 → Trace 一键跳转
  Loki 日志行 trace_id=xxx → Grafana derivedFields → URL → SkyWalking UI
```

### 5.2 数据不是双写，是各存各的

```
正确理解:
  日志 → Loki (5Gi)   ← 按 label+时间 查询，轻量索引
  Trace → ES (10Gi)   ← 按字段聚合/过滤，倒排索引
  指标 → Prometheus TSDB (50Gi) ← 时间序列数据库

错误理解:
  日志 → ES  ❌  (没有 Logstash/Filebeat，日志不进 ES)
  Loki 没有数据库  ❌  (Loki 有 BoltDB 索引 + 压缩 chunck)
```

### 5.3 存储物理分布

| 数据 | PVC | 物理节点 | 路径 |
|------|-----|---------|------|
| 指标 TSDB | prometheus-server (50Gi) | worker1 | `/opt/local-path-provisioner/pvc-559a947c...` |
| 仪表盘 | grafana (10Gi) | worker1 | `/opt/local-path-provisioner/pvc-476b6d16...` |
| 日志 | loki-storage (5Gi) | worker2 | `/opt/local-path-provisioner/pvc-7889a08c...` |
| Trace | elasticsearch-data (10Gi) | worker1 | `/opt/local-path-provisioner/pvc-411127da...` |
| 静默 | storage-alertmanager-0 (50Mi) | — | — |
