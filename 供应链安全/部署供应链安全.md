# 第 24 项 · 供应链安全：镜像签名（Cosign）+ SBOM + Kyverno（策略即代码引擎） 验签闭环

> 目标：实现「**只允许运行经 cosign（镜像签名工具） 签名的镜像**」的准入闭环——签名镜像放行，未签名镜像被拒绝。
> 配套能力：用 Syft 生成 SBOM（软件物料清单），形成「签名可验证 + 物料可追溯」的供应链安全基线。

---

## 1. 架构概览

```
                ┌──────────────────────────────────────────────┐
  开发者/CI ──> │  h1:5000 本地 HTTPS 仓库 (distribution)       │
   cosign sign  │   - signed/nginx:1.25   (已签名)              │
                │   - unsigned/busybox:1.36 (未签名, 对照用)    │
                └───────────────▲───────────────────┬──────────┘
                                │ 拉取签名/镜像       │ 验签时拉取签名
                                │                    │
                        worker 节点 containerd   Kyverno 准入控制器 (w1)
                        (已信任仓库自签 CA)        │ verifyImages + require-signed-images
                                                 │ 公钥校验 + rekor.ignoreTlog
                                                 ▼
                                          未签名 → 拒绝 (Enforce)
                                          已签名 → 放行 (Pod Running)
```

- **本地仓库**：在 h1 用 `goharbor/registry-photon:v2.9.0`（distribution）起一个**无认证 HTTPS** 仓库，端口 `192.168.1.61:5000`。
  - 选「无认证」是为绕开 Harbor（私有镜像仓库） 的 SSRF 防护坑（见踩坑 ①）；仓库自签证书，集群侧信任其 CA。
- **Kyverno**：`verifyImages` 规则 + `ClusterPolicy require-signed-images`，`validationFailureAction: Enforce`。
- **cosign v2.2.3 / syft v1.46.0**（装在 h1）。

## 2. 签名与 SBOM（在 h1 执行）

脚本见 `cosign-sbom/sign-and-sbom.sh`，要点：

```bash
export COSIGN_PASSWORD=cosign
cosign generate-key-pair                       # 生成 cosign.key / cosign.pub
echo y | cosign sign --key cosign.key 192.168.1.61:5000/signed/nginx:1.25
cosign verify --key cosign.pub 192.168.1.61:5000/signed/nginx:1.25
syft 192.168.1.61:5000/signed/nginx:1.25 -o spdx-json > sbom-nginx.spdx.json
```

- 验签输出中 `docker-reference` 应为 `192.168.1.61:5000/signed/nginx`，即身份与引用一致。
- 策略里内联的 `publicKeys` 就是这里的 `cosign.pub`（见 `策略即代码-Kyverno/policies/require-signed-images.yaml`）。

## 3. Kyverno 验签闭环（在 m1 执行）

### 3.1 让准入控制器信任本地仓库 CA（关键）

本版本 Kyverno 读取 registry 配置的 ConfigMap（配置字典） 名为 **`config`**（namespace `kyverno`），其 `registries` 字段里的 `ca` 在本环境的 **cosign 验签拉取路径上并未被注入 TLS 信任池**，表现为对 `192.168.1.61:5000` 始终报
`x509: certificate signed by unknown authority`，且回退 HTTP 后收到 `400 Bad Request`。

可靠解法：把本地仓库 CA **注入准入控制器 Pod（容器组） 的系统信任库**（initContainer 追加到 `/etc/ssl/certs/ca-certificates.crt`）。

```bash
# kyverno-verify/ca-inject.sh（需在 m1 以 root + KUBECONFIG=/etc/kubernetes/admin.conf 执行）
kubectl -n kyverno apply -f kyverno-localreg-ca.yaml
kubectl -n kyverno patch deploy kyverno-admission-controller --type=json -p='[
  {"op":"add","path":"/spec/template/spec/volumes/-","value":{"name":"localreg-ca","configMap":{"name":"kyverno-localreg-ca"}}},
  {"op":"add","path":"/spec/template/spec/volumes/-","value":{"name":"ca-trust","emptyDir":{}}},
  {"op":"add","path":"/spec/template/spec/initContainers/-","value":{"name":"trust-localreg","image":"192.168.1.61:5000/unsigned/busybox:1.36","command":["/bin/sh","-c"],"args":["cp -r /etc/ssl/certs/. /ca-trust/ 2>/dev/null; cat /localreg-ca/localreg.crt >> /ca-trust/ca-certificates.crt"],"volumeMounts":[{"name":"localreg-ca","mountPath":"/localreg-ca"},{"name":"ca-trust","mountPath":"/ca-trust"}]}},
  {"op":"add","path":"/spec/template/spec/containers/0/volumeMounts/-","value":{"name":"ca-trust","mountPath":"/etc/ssl/certs"}},
  {"op":"add","path":"/spec/template/spec/containers/0/env/-","value":{"name":"COSIGN_EXPERIMENTAL","value":"0"}}
]'
kubectl -n kyverno rollout restart deploy kyverno-admission-controller
```

### 3.2 策略

`策略即代码-Kyverno/policies/require-signed-images.yaml` 核心：

```yaml
verifyImages:
- imageReferences: ["192.168.1.61:5000/*"]   # 本仓库所有镜像必须签名
  attestors:
  - entries:
    - keys:
        publicKeys: |
          -----BEGIN PUBLIC KEY-----
          ...（cosign.pub 内容）...
          -----END PUBLIC KEY-----
        rekor:  { ignoreTlog: true }          # 离线：跳过 Rekor 透明日志
        ctlog:  { ignoreSCT: true }           # 离线：跳过 SCT 校验
  required: true
  mutateDigest: true
  verifyDigest: true
```

`rekor.ignoreTlog` / `ctlog.ignoreSCT` 必须加（见踩坑 ③）。

### 3.3 闭环验证结果（实测）

测试 Pod（均带 `livenessProbe`/`readinessProbe` 与 `securityContext.privileged: false`，以不被其它策略误伤）：

| 用例 | 镜像 | 结果 |
|------|------|------|
| 签名镜像 | `192.168.1.61:5000/signed/nginx:1.25` | ✅ **Running**（1/1，调度到 worker2） |
| 未签名镜像 | `192.168.1.61:5000/unsigned/busybox:1.36` | ❌ 被 `require-signed-images` 拒绝：`no signatures found` |

`kubectl -n signature-demo get pod` 实测输出：
```
NAME           READY   STATUS    RESTARTS   AGE   IP          NODE
signed-nginx   1/1     Running   0          15s   10.0.4.27   worker2
```
未签名 Pod 创建被准入拒绝，命名空间内只留下签名 Pod —— **闭环成立**。

## 4. 踩坑实录（重要）

1. **Harbor 的 SSRF 防护导致验签失败**：Kyverno v1.13+ 的 go-containerregistry 拒绝跟随 Harbor 返回的私有 IP `bearer` realm（`invalid realm ... is a private or link-local address`）。Harbor 的 `hostname` 固定为 `192.168.1.61` 不可轻易改（波及 Cilium（基于 eBPF 的 CNI/网络方案）/MinIO/Falco 等全部靠它拉镜像的组件）。→ 改用 h1 上独立的**无认证 HTTPS 本地仓库**做闭环演示。
2. **Kyverno registry 配置位置与 `ca` 失效**：registry 配置在 `kyverno` namespace 的 **`config`** ConfigMap（`registries` 键），不是 `kyverno` CM 里的 `config` 键（那里面曾被 PowerShell 把 `registries:` 损坏成 `writtenes:`）。即便 `config` CM 的 `ca` 写对了，本版本 cosign 验签拉取**也不认这个 `ca`** → 必须走「CA 注入 Pod 系统信任库」（第 3.1 节）。
3. **离线环境的 Rekor/TUF 报错**：cosign 验签默认会初始化 Rekor/TUF 客户端，去 `tuf-repo-cdn.sigstore（软件签名框架）.dev` 下载 `13.root.json`，离线环境直接失败（`x509: certificate signed by unknown authority` 或 `lookup ... server misbehaving`）。→ 策略里加 `rekor.ignoreTlog: true` 与 `ctlog.ignoreSCT: true`。
4. **cosign v2.2.3 只认自己的密钥格式**：`openssl` 生成的 `EC PRIVATE KEY` / `PRIVATE KEY` / `COSIGN PRIVATE KEY` 全部被拒，仅接受 `cosign generate-key-pair` 产出的 `ENCRYPTED SIGSTORE PRIVATE KEY`。→ 直接用 `cosign generate-key-pair`（用 `COSIGN_PASSWORD` 环境变量实现非交互）。
5. **Pod 字段错误**：`spec.securityContext.privileged` 非法（`privileged` 只能在容器级 `spec.containers[].securityContext`）。否则 Pod 创建报 `unknown field "spec.securityContext.privileged"`。
6. **PowerShell CLIXML 噪音**：Windows 宿主机经 SSH 执行含管道/引号/换行的命令会被吞输出或解析错。→ 一律「写脚本文件 → `scp` 到目标 → `echo '123' | sudo -S bash /tmp/x.sh > /tmp/x.log` → 回读日志」。
