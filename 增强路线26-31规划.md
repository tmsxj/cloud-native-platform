# 后续增强路线 26~31 规划（云原生进阶研究方向）

> 📅 规划日期: 2026-07-12
> 🎯 定位: **研究方向清单，非短期待办**。第 20~25 项已全部落地并验证，本规划用于明确"作品集之外的云原生进阶探索方向"，短期内不实施。
> 📌 现状基线: 单 kubeadm 集群（3 master + 2 worker，Cilium CNI + Hubble，Harbor 镜像仓库，Prometheus/Grafana/Loki/OpenTelemetry 可观测性栈，Kyverno/Falco/cosign 安全链，Linkerd+Istio 双服务网格，ArgoCD/Argo Rollouts 三种 CI/CD 路线）。

## 26 · 多集群与联邦治理（Multi-Cluster）

**目标**：突破单集群边界，实现跨集群服务互通、全局流量调度、多活/容灾。

**技术选型（研究对比）**：
- **Cilium ClusterMesh**：基于已部署的 Cilium，跨集群 Pod 直连（同网络平面），几乎零新增组件，最适合先验证。
- **Karmada**：多集群编排调度器，统一 API 下发到多个成员集群，适合"一对多"管控。
- **Submariner**：跨集群 L3 连通（Gateway 模式），对接非 Cilium 集群。

**研究要点**：
- 跨集群 Service 发现（Global Service / ClusterIP 跨集群可达）
- 全局负载均衡与就近访问
- 集群级故障切换 runbook（主集群宕机时流量/工作负载迁移）
- 与现有 Kyverno/Falco 策略的多集群下发

**预期产出**：双集群部署 + 服务互通演示 + 容灾切换 runbook（文档为主，非长期常驻）。

---

## 27 · 事件驱动弹性 KEDA（Serverless 化）

**目标**：让工作负载按"事件"而非仅按"CPU/内存指标"伸缩，并支持缩容到 0。

**技术选型**：KEDA（Kubernetes Event-Driven Autoscaling）+ `ScaledObject`/`ScaledJob`，支持 Kafka / Redis / RabbitMQ / Cron / 自定义指标等 60+ 事件源。

**研究要点**：
- KEDA 与 HPA 的协作关系（KEDA 在 HPA 之上生成外部指标，不直接替代 HPA）
- 缩容到 0 的冷启动延迟与就绪探针配合
- 事件源对接（以消息队列 demo 为主）
- 与 Argo Rollouts（第 31 项）协同做事件驱动的渐进式发布

**预期产出**：一个"消息积压 → 自动扩容 → 消费完 → 缩到 0"的可演示 demo。

---

## 28 · Gateway API 标准化落地

**目标**：用 Ingress 的继任标准 `gateway.networking.k8s.io`（Gateway/HTTPRoute/GRPCRoute）替代功能受限的传统 Ingress。

**技术选型**：Gateway API CRD + 数据面（复用 Istio ingress-gateway，或 Cilium Gateway API 支持）。

**研究要点**：
- 与传统 Ingress 的能力对比（按权重引流、Header 路由、跨命名空间路由、灰度）
- 多租户 Gateway（不同团队共享数据面、独立路由策略）
- 与现有双服务网格（Linkerd/Istio）的入口整合
- TLS Passthrough / SNI 路由

**预期产出**：Gateway API 替代 Ingress 的对照示例 + 灰度引流示例。

---

## 29 · Kubernetes 排障与性能调优（内功）

**目标**：沉淀一套可复用的故障定位方法论与调优 checklist（偏文档/实战，非新组件部署）。

**研究要点**：
- **内核/运行时**：cgroup v2 层级与资源约束、内核参数（net/ipc/vm）、HugePages
- **etcd**：性能瓶颈、碎片、defrag、配额与监控
- **网络栈**：基于已有 eBPF 能力做网络抓包/延迟分析（Cilium/Hubble 已有基础）
- **调度与驱逐**：kubelet 压力驱逐、OOM/Eviction 机制、节点资源预留
- **故障定位决策树**：Pod 起不来 / 网络不通 / 性能抖动 的标准排查路径

**预期产出**：《K8s 排障决策树 + 性能调优 Checklist》文档 + 若干调优前后对比实验。

## 30 · SLO 工程与告警降噪

**目标**：从"能看监控"升级到"可用性工程"——用 SLO/错误预算驱动发布与告警。

**技术选型**：Prometheus 多窗口多燃尽率告警（burn-rate alerts）+ Alertmanager 分组/抑制/静默 + Grafana SLO 面板（可结合 OTel 已部署栈）。

**研究要点**：
- SLI/SLO 定义（请求成功率、延迟、饱和度等黄金信号）
- 错误预算（Error Budget）消耗策略与"预算烧完即停发"
- 告警降噪：去重、分组、抑制、值班路由（避免告警风暴）
- 与现有 Prometheus/Grafana/Loki 打通，不引入新数据源

**预期产出**：一个核心服务的 SLO 定义 + 错误预算策略 + 降噪后的告警路由配置。

## 31 · GitOps 进阶与渐进式交付深化

**目标**：把已有的"三套 CI/CD 方案"从演示清单升级为端到端 GitOps 闭环，并引入指标驱动的自动发布决策。

**技术选型**：ArgoCD `ApplicationSet`（多环境/多集群批量同步）+ Argo Rollouts 蓝绿/金丝雀 + `Analysis`（基于 Prometheus 指标自动判稳）。

**研究要点**：
- GitOps 闭环：Git 提交 → ArgoCD 自动同步 → 集群状态收敛
- 渐进式交付：金丝雀按流量比例递进，每步用 Analysis 校验指标（错误率/延迟）
- 自动继续/回滚决策（指标不达标自动回滚，达标自动推进）
- 多环境（dev/staging/prod）用 ApplicationSet 统一管理

**预期产出**：一个从 Git 提交 → 自动同步 → 金丝雀发布 → 指标判稳自动继续/回滚 的完整闭环 demo。

## 实施建议（未来若启动）

1. **优先级**：26（多集群）和 31（GitOps 深化）最能体现云原生进阶深度，建议作为"首批研究"；29（排障内功）性价比最高但最吃经验。
2. **依赖顺序**：28 可先于 31（先统一入口标准再谈交付）；27 相对独立可随时插队。
3. **资源约束**：多集群（26）需要第二套集群或轻量 kind/k3s 集群，需评估机器资源；其余 5 项均可在现有单集群内研究验证。
4. **与聚焦模式关系**：本规划不要求"激活全家桶"，多数方向只需控制面 + Cilium + 可观测性即可开展。
